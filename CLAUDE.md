# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Christopher Hotchkiss's personal site / CRM. A single self-contained Rust binary that ships as a macOS `.app` with a tray icon, serves the site over HTTPS, and self-manages its own DNS (Cloudflare) and TLS certificates (ACME via Let's Encrypt). User roles: `Anonymous` (read-only) / `Registered` (read-only, logged in) / `Admin` (edit). The very first user to register is automatically promoted to `Admin` (enforced inline in `UserDao::create`).

## Common commands

- `bacon` â€” default dev loop. Runs `cargo run -- data/config.json`, kill-restarts on changes in `assets/`, `styles/`, `templates/`. Press `c` for clippy.
- `cargo build` / `cargo run -- data/config.json` â€” direct equivalents.
- `cargo test` â€” runs the suite. SQLx unit tests use `#[sqlx::test(migrator = "crate::db::database_handle::MIGRATOR")]` (isolated SQLite per test). `tests/` integration tests (`server.rs`, `web.rs`) spin up the web layer in-process via `hotchkiss_io::test_support::spawn_test_server()` (no IP/DNS/ACME coordinator; plain HTTP on `127.0.0.1:0`) and hit it with `reqwest`. A debug-only `POST /test/login?role=Admin|Registered` seam (`web/features/test_login.rs`, `#[cfg(debug_assertions)]` â€” absent from release) lets them reach role-gated routes without the WebAuthn dance.
- Browser e2e run as part of **plain `cargo test`** (no longer `#[ignore]`d â€” needs Google Chrome installed; `cargo test --test e2e_browser` to run just them). Pure Rust via `chromiumoxide` (CDP): drives headless Chrome against `spawn_test_server`, using a CDP **virtual authenticator** to complete the real passkey registration ceremony, then checks the `/admin/analytics` auth gate. They **serialize on a shared `E2E_LOCK` mutex** â€” each launches its own Chrome + WebAuthn authenticator, and running the passkey ceremonies concurrently races (blows the 20s registration deadline). **No test may be `#[ignore]`d** â€” a `lint_no_ignored_tests` guard test scans `src/` + `tests/` and fails if the attribute reappears (ignored tests rot silently â€” three e2e tests drifted out of sync with the UI that way). A test that can't always run gets a **Cargo feature gate** instead: the iOS-Simulator e2e (`tests/e2e_ios.rs`, WebKit rendering checks) is gated behind `--features ios_e2e` (boots a sim, 10â€“60s, needs Xcode) and run on the mini, not in the default suite.
- `cargo test <test_name>` â€” single test (e.g. `cargo test roundtrip`).
- `npm run format` â€” Prettier + Tailwind plugin formatting (templates / styles / config).
- `./build/macos/build.sh [--profile beta|prod]` â€” builds an ad-hoc-signed app bundle (no notarization, no `.pkg`; the DEPLOY re-signs it with a Developer ID via the GUI-session signer agent for durable Full Disk Access â€” Phase CP, see the deploy section). `--profile` (default `prod`) picks the bundle name + identifier so beta and prod coexist in `/Applications` and LaunchServices: `prod` â†’ `Hotchkiss-IO.app` / `io.hotchkiss.web`, `beta` â†’ `Hotchkiss-IO-Beta.app` / `io.hotchkiss.web.beta`. Honors `CARGO_TARGET_DIR`; writes `$CARGO_TARGET_DIR/<name>.app` and prints `BUILT_APP=<absolute-path>` + `PROFILE=<profile>` on stdout. Resolves `VERSION` from `$VERSION` â†’ `$GITHUB_REF_NAME` â†’ `git describe`, falling back to `0.0.0-dev` â€” on the mini the post-receive extracts a `.git`-less tree, so CFBundleVersion lands on `0.0.0-dev` (cosmetic; the runtime/log/tray version comes from `CARGO_PKG_VERSION`, so bump `Cargo.toml` to match each tag â€” single-source fix is backlogged). The post-receive hook calls this per profile.

Deployment is push-to-deploy with an **inverted flow** (Phase 12): one bare repo at `~/repos/hotchkiss-io.git` on the Mac mini, whose `post-receive` hook (`build/macos/post-receive`) **dispatches by pushed ref** â€” `git push origin main` â†’ **beta** (bleeding edge), and `git push origin vX.Y.Z` (a `v*` tag) â†’ **prod** (deliberate promotion). For each deploying ref the hook archives the tree into a per-profile work dir, runs `build/macos/build.sh --profile <profile>`, atomically swaps the new `.app` into `/Applications`, and **gracefully restarts** the matching LaunchAgent â€” `launchctl kill TERM` (NOT `kickstart -k`/SIGKILL, which made macOS file a "quit unexpectedly" crash report on the mini every deploy, Phase BY); `KeepAlive=true` relaunches it onto the swapped bundle (after up to `ThrottleInterval`, kept as a crash-loop backstop), self-bootstrapping if it isn't loaded. A failed build leaves the running app untouched, and each ref deploys in an isolated subshell so one ref's failure can't skip a sibling. **Concurrency + fail-loud (Phase CO incident):** each profile's deploy holds a per-profile `mkdir` lock (macOS has no `flock`; 600s stale-break, released via an EXIT trap with the path baked in â€” a lazy `$local` is an unbound var under `set -u` and never releases) so OVERLAPPING pushes of the SAME profile serialize â€” three concurrent beta pushes once raced the `/Applications` swap, nested the `.app` into `.app.prev`, left no live bundle (beta down) while a racer still printed "deployed" + exited 0. The swap now also asserts a valid bundle (`Contents/MacOS/hotchkiss-io` executable) BEFORE the restart and ABORTS non-zero if absent â€” `set -e` catches an `mv` error but NOT an `mv` that nested into an existing dir (exit 0), the exact silent-corruption path. **The prodâ†’beta media rsync carries `--max-size=2G`** so the ephemeral beta skips prod's multi-GB test assets (a 78 GB media-test file was otherwise re-copied on every beta deploy); prod + B2 keep the big files. **Releasing to prod:** bump `Cargo.toml`'s `version`, commit, `git push origin main` (lands on beta), then `git tag vX.Y.Z && git push origin vX.Y.Z`. Two LaunchAgents run side-by-side â€” prod `io.hotchkiss.web` on `:80`/`:443`, beta `io.hotchkiss.web.beta` on `:8080`/`:8443`. `origin` is the mini (the deploy remote); the public `github` mirror (`github.com/chotchki/hotchkiss-io`) was **re-added 2026-06-29** once the docs were de-personalized + the history rewritten to purge personal career context (Phase CM). Publishing is manual â€” `git push github main` + `git push github $(git tag -l 'v*')`; the mini stays the deploy path and nothing auto-pushes to GitHub. (Note: the mini's pre-CM `v*` tags still point at the OLD pre-rewrite lineage â€” they were deliberately NOT force-updated, since pushing 35 moved tags to the mini would trigger 35 prod deploys; GitHub carries the rewritten tags. New releases tag from the rewritten `main`, so both stay clean going forward.) **Code signing (Phase CP):** `build.sh` ad-hoc-signs (so the binary runs), then the deploy **re-signs with chris's Developer ID** â€” a STABLE identity so the macOS **Full Disk Access** (TCC) grant survives deploys (ad-hoc's per-build cdhash dropped FDA on every push â†’ the external `/Volumes/MediaStorage4` media started *hanging*). `codesign` CANNOT reach the keychain from the non-GUI sshd hook (`errSecInternalComponent`; `launchctl asuser` into the GUI session needs root), so signing is delegated to a **GUI-session LaunchAgent** (`io.hotchkiss.signer` / `build/macos/sign-agent.sh`, WatchPaths a request dir): `post-receive`'s `sign_with_devid` drops the built app's path and waits for the agent's result **before the swap** â€” a sign failure ABORTS the deploy (never ships an ad-hoc binary that would silently drop FDA). NO hardened runtime (`--options runtime`): we don't notarize, and Dev-ID + hardened runtime enables library validation that `OS_REASON_CODESIGNING`-kills the app at launch; TCC keys on the Team ID regardless. **FDA is a ONE-TIME grant** per app (System Settings â†’ Privacy & Security â†’ Full Disk Access), then it sticks across every deploy. The swap also stopped `rm`ing `.app.prev` in-deploy (unlinking the exiting old process's mmap'd executable made the kernel `OS_REASON_CODESIGNING`-kill it â†’ a "quit unexpectedly" popup, the other half of the Phase BY goal); the NEXT deploy's top-`rm` reclaims it. **Mini setup** â€” cert import + `security set-key-partition-list` + `build/macos/install-signer-agent.sh` (scp's the agent/hook to the mini + self-tests signing through the agent) â€” is in `build/macos/SETUP.md` Â§9. Notarization / `.pkg` stay retired (signing-IDENTITY only, for TCC stability).

**Beta instance** (`beta.hotchkiss.io:8443`) is public (grey-cloud A record that beta's own `DnsProviderService` tracks to the public IP, reached via a router `:8443` forward) and serves its **own** LE-prod cert â€” beta deploys as a release build, so the cert is natively trusted (the iPhone installs the PWA with no profile). Every `main` push **snapshots prod's `database.sqlite` into beta** (`snapshot_prod_db_into_beta` in the hook, via `sqlite3 .backup` of the live prod DB): it scrubs `crypto_keys` EXCEPT id 2 (beta regenerates its session signing key id 1 but KEEPS the media-URL HMAC key, so its `url_key`s match prod's), `tower_sessions`, `request_log` (visitor privacy), and `instant_acme_domains` (prod's LE account key); **1-for-1 rsyncs prod's media into beta's media** (content-addressed, `--ignore-existing` so incremental): prod's and beta's `media_paths` are MATCHING ordered roots, so `prod_roots[i]â†’beta_roots[i]` keeps each root on the SAME class of drive â€” beta's root[0] is `/Volumes/MediaStorage4/website_beta`, so the big external media stays external (consolidating prod's multi-drive media onto beta's one internal dir was the v0.0.83 boot-disk-fill, fixed in CP) + a drive-present guard skips an unmounted root; beta's config sets `media_paths`/`backup_path` under `io.hotchkiss.web.beta/`, since `app_support` is hardcoded to `io.hotchkiss.web`; **preserves** beta's own cert + ACME account (captured/restored, so beta never re-orders against the 5/week LE duplicate-cert limit); and **carries prod's `users`/passkeys** so chris's prod passkey authenticates on beta â€” beta's `webauthn_rp_id` is the registrable parent `hotchkiss.io` (set in beta's config), and the WebAuthn `rp_origin` includes the non-default `:8443` port. Beta data is therefore ephemeral by design: any beta-only edit is blown away on the next `main` push.

## Build-time machinery (build.rs)

`build.rs` does four things that surprise people:

1. **Creates an ephemeral SQLite schema DB at `$OUT_DIR/schema.db`** and runs `src/db/migrations/` against it, then writes `DATABASE_URL` into `.env` and exports it to rustc. This is what lets `sqlx::query!` / `query_as!` macros type-check at compile time. **If you add a migration or change a query, you may need `cargo clean -p hotchkiss-io` (or delete the OUT_DIR `schema.db`) for sqlx macros to re-validate.**
2. **Downloads a pinned Tailwind CLI** (`TAILWIND_VERSION` in `build.rs`, currently `v4.3.0`, arm64 macOS standalone binary) into `OUT_DIR` (cached under a version-keyed filename, so bumping the pin re-downloads) and compiles `styles/tailwind.css` â†’ `assets/styles/main.css`. The compiled CSS is gitignored. (DaisyUI used to be downloaded here too but was never wired into `tailwind.css` â€” removed 2026-05; the site is styled with hand-rolled Tailwind utilities.)
3. **Generates the inline-SVG icon set** (Phase CN): reads the vendored Font Awesome Free 6.7.2 solid SVGs in `build/svg-icons/solid/` (kept OUTSIDE `assets/`, so `rust-embed` doesn't ship them) and emits `templates/partials/icons.html` â€” one parameterless askama macro per icon, called as `{% call icons::pen() %}` â†’ inline `<svg class="icon" fill="currentColor">` (`.icon` is `height:1em;width:auto;fill:currentColor` in `tailwind.css`, so it inherits text size/color; wrap a call in a sized `<span>` for a bigger/recolored icon). **FontAwesome is GONE** â€” the 155 KiB `fa-solid-900.woff2` + ~18 KiB CSS were the single biggest LCP critical-path cost (PSI render-blocking ~1.9s). **To add a UI icon: drop its SVG in `build/svg-icons/solid/<name>.svg` and call `icons::<name_with_underscores>()`** â€” a typo'd macro is a compile error (no un-vendored icon sneaks in, so there's no template scan). WRITE-IF-CHANGED so an unchanged regen doesn't bump the mtime + cascade a recompile (the partial lands under the `templates` rerun-trigger); the generated `icons.html` is gitignored. The STL-viewer loading spinner is a self-contained `.stl-spinner` CSS ring (no icon, no dep on Tailwind scanning JS).
4. Re-runs only when `assets/scripts`, `templates`, `migrations`, or `build/svg-icons` change.

`assets/` and `templates/` are bundled into the binary at compile time via `rust-embed` (`web/static_content.rs`) and `askama` respectively â€” there are no loose static files at runtime besides the SQLite database.

## Runtime architecture

`src/main.rs` â†’ `lib::real_main` â†’ `tray-wrapper` (macOS menu-bar anchor) â†’ `create_server` closure â†’ `ServiceCoordinator`.

The coordinator (`src/coordinator/service_coordinator.rs`) wires four long-lived tokio tasks connected by `broadcast` channels:

```
IpProviderService â”€â”€IPsâ”€â”€â–¶ DnsProviderService (Cloudflare API)
                                 â”‚
AcmeProviderService â”€â”€RustlsConfigâ”€â”€â–¶ EndpointsProviderService (HTTP/HTTPS axum servers)
```

- **IpProviderService** polls `https://1.1.1.1/cdn-cgi/trace` hourly (parsing the `ip=` line) and broadcasts the public IP. Connecting to the IPv4 literal forces a v4 path. In `debug_assertions` builds it forces `127.0.0.1` so dev never touches the public DNS.
- **DnsProviderService** updates Cloudflare A records when the IP differs from DNS.
- **AcmeProviderService** loads or orders a Let's Encrypt cert (DNS-01 via Cloudflare), persists it in SQLite (`certificates` table), broadcasts a `RustlsConfig`, and refreshes every 6h. Serves **HTTP/2** (ALPN `h2`, h1.1 fallback; no HTTP/3) â€” the ALPN policy + the axum-server-needs-ALPN-or-it-silently-stays-h1 gotcha live in `build_server_config` and its unit test.
- **EndpointsProviderService** waits for a `RustlsConfig` then starts both an HTTPâ†’HTTPS redirect server on `:80` and the real axum app on `:443` (served with `into_make_service_with_connect_info::<SocketAddr>()` so handlers/middleware can see the client IP). It also owns the `tower-sessions-sqlx-store` session GC task, a daily `request_log` prune task (90-day retention), and a **daily DB backup task** (`coordinator/backup.rs`): each tick `VACUUM INTO`s a dated snapshot (`database-YYYY-MM-DD.sqlite`) under `Settings::backup_path`, then prunes to a rolling 7 most-recent. The backup loop never returns and matches+logs every fallible step (`tracing::error!`) so a backup failure can't bubble out and trip the coordinator's `try_join!` (which would take the whole app down) â€” a bad day self-heals on the next tick. Backups exist only on disk; Backblaze covers off-site.

If any task returns, `tokio::try_join!` fails the whole coordinator â€” there is no auto-restart.

## Web layer (`src/web/`)

- **Routing** is composed in `web/router.rs`: `/` redirects to the first content page, `/login`, `/pages`, `/projects`, `/blog`, `/admin`, `/media` are nested routers (`/attachments` was retired in BZ.8), and `static_content()` is merged in. The headline numbers DEFAULT to All (factual) with an always-visible **All/Humans/Bots 3-chip** (`?audience=`, threaded through the count queries as a `?N='All' OR ua_class=?N` OR-of-constant so the `ts` index still bounds the scan; - **Templating**: askama templates in `templates/`, served via the `HtmlTemplate<T>` wrapper. - **Auth**: WebAuthn / passkeys via `webauthn-rs` (discoverable credentials, conditional UI). State machine in `AuthenticationState`: `Anonymous â†’ AuthOptions â†’ Authenticated`, or `Anonymous â†’ RegistrationStarted â†’ Authenticated`. Registration first creates the `UserDao` with `Role::Anonymous`; the SQL `INSERT` then conditionally upgrades to `Admin` (first user) or `Registered`.
- **API-key auth (Phase CA)**: an `Authorization: Bearer hio_â€¦` key authenticates as its user, **delegating that user's role** (full delegation â€” a key minted by an Admin has Admin). The `api_key_auth` middleware (`web/middleware/api_key_auth.rs`, `from_fn_with_state`, layered OUTER to the authz layer) resolves the key and injects an Authenticated `SessionData` into the request; the `SessionData` extractor reads that injection FIRST, so the existing fail-closed layer + every handler see the key's user with ZERO handler changes (no prebuilt DB-backed-key middleware fits â€” `tower_http::ValidateRequestHeaderLayer` is single-static-token + sync only). Keys are user-managed at `/admin/api-keys` (generate â†’ plaintext shown ONCE, list with last-used, revoke instantly â€” scoped so you only see/revoke your own). Stored as **HMAC-SHA256(server pepper = `crypto_keys` id 3, key)** hex â€” the key itself is never stored, so a DB leak can neither recover nor verify it; live until revoked (`api_keys` table, migration `0016`; `hio_<base64url>` from 32 random bytes). **Env-specific:** the beta snapshot scrubs `crypto_keys` id 3 (only id 2 is preserved), so prod keys don't authenticate on beta and the carried `api_keys` rows are inert there â€” generate a beta-specific key if you need one.
- **User management (Phase CC)**: `/admin/users` (`web/features/admin/users.rs`, admin-gated by the `/admin` nest's `require_admin`) lists every user (role + passkey count = `json_array_length(keys)` + live API-key count) and promotes/demotes between `Registered`â†”`Admin` (`POST /admin/users/{id}/role`) or deletes (`DELETE /admin/users/{id}`, which wipes the user's `api_keys` first â€” FK â€” then the row, in a tx; passkeys live in the `keys` column and go with it). **The last admin is protected** from demote/delete (no lockout) â€” enforced in the handlers (`409`) AND hidden in the UI; self-actions are otherwise allowed. There's still **no create-user** path: users self-register via passkey (first registrant â†’ Admin). **Live enforcement** â€” a cookie session stores a SNAPSHOT of the user (`Authenticated(UserDao)`, role included), so a role change / delete wouldn't bite a live session until logout/expiry. The `refresh_session_role` middleware (`web/middleware/refresh_session_role.rs`, `from_fn_with_state`, layered INNER to `api_key_auth` / OUTER to the authz layer) re-loads the cookie session's user by id each request and injects a refreshed `SessionData` (current role) â€” or Anonymous if the user was deleted â€” so demote/delete take effect IMMEDIATELY (a Bearer key already resolves fresh, so the middleware no-ops for it). One extra cheap SQLite lookup per authenticated request.
- **Cookie hardening (Phase CC)**: the session cookie â€” the ONLY cookie the app sets (tower-sessions) â€” is explicitly `HttpOnly` + `SameSite=Lax` + `Secure` (release only; `with_secure(!debug)` keeps it off for the plain-HTTP test harness / dev). `HttpOnly` is tower-sessions' default, set explicitly so it can't silently regress; an integration test asserts `Set-Cookie` carries `HttpOnly` + `SameSite`.
- **Authorization is one fail-closed layer** (`web/middleware/require_admin_for_mutations.rs`, Phase E): **GET/HEAD/OPTIONS are public site-wide; every other method requires an authenticated Admin BY DEFAULT.** It's a default-DENY allowlist of safe methods (not a deny-list of known-mutating verbs), so a newly-added route or an exotic verb (e.g. the `/pages/preview` PATCH) is gated automatically â€” the secure default protects against the per-handler-omission bug it replaced (v0.0.49: an anonymous caller could mutate `/pages`). The only exceptions are an EXACT `(path, method)` allowlist of the anonymous WebAuthn ceremony POSTs (`/login/finish_authentication`, `/login/finish_register`) + the debug `/test/login` seam. Wired in `router.rs` **inner to the session layer** (so `SessionData` is loaded; it defaults to `Anonymous` â†’ a clean 403, not a panic). The **`/admin` nest keeps its own `require_admin` layer** (`web/middleware/require_admin.rs`) â€” it gates a *non-public* GET (`/admin/analytics`) that the GET-public default would otherwise expose; the two are complementary. **No per-handler authz checks remain** â€” the layer is the single source of truth for mutations. (`list_page_attachments` GET was made public in Phase F: the attachment *content* is already unauthenticated, so the listing leaks nothing.)
- **Errors**: handlers return `Result<Response, AppError>`. `AppError` wraps `anyhow::Error`, logs with a UUID trace id, and returns a 500 with that id to the client.
- **`AppError` swallows status info** â€” if a handler needs to return e.g. a 404 or 403, return `Ok((StatusCode::X, "msg").into_response())` instead of bubbling via `?`. This is a deliberate pattern, see existing handlers for examples.

## Configuration

`Settings` is loaded from JSON, in this order:

1. First CLI arg, if present (this is what `bacon` and dev use: `data/config.json`).
2. Otherwise `~/Library/Application Support/io.hotchkiss.web/config.json` on macOS, `$HOME/...` elsewhere.

Required fields: `cloudflare_token`, `domain`. Optional path fields â€” `database_path`, `log_path`, `cache_path`, `backup_path` â€” default to `~/Library/Application Support/io.hotchkiss.web/data/database.sqlite`, `~/Library/Logs/io.hotchkiss.web`, `~/Library/Caches/io.hotchkiss.web`, and `~/Library/Application Support/io.hotchkiss.web/backups` respectively when omitted (see `Settings::resolve` in `src/settings.rs`). `backup_path` is where the daily DB snapshots land (created on first backup if missing). The on-disk JSON is deserialized into a private `RawSettings`; unknown keys are silently ignored. The `data/` directory in the repo is gitignored and contains the dev config + SQLite database.

The macOS app is **not sandboxed** (the `com.apple.security.app-sandbox` entitlement was dropped, 2026-05) â€” `NSHomeDirectory()` returns the real home, so the paths above are literal. On the production Mac the app runs as a LaunchAgent (`build/macos/io.hotchkiss.web.plist`) in the user GUI session, which keeps the tray icon alive; macOS Mojave+ lets the non-root process bind ports 80/443 because axum binds `INADDR_ANY`.

## Things to watch out for

- **Vendored OpenSSL**: `openssl` is pinned with `features = ["vendored"]` to avoid system-OpenSSL packaging issues.
- **Ports default to 80/443 but are configurable** (`http_port` / `https_port` in `Settings`, since Phase 11; beta runs 8080/8443) â€” running locally as non-root on 80/443 will fail to bind, so set high ports in your dev config or accept the bind failure.
