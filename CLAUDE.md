# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Christopher Hotchkiss's personal site / CRM. A single self-contained Rust binary that ships as a macOS `.app` with a tray icon, serves the site over HTTPS, and self-manages its own DNS (Cloudflare) and TLS certificates (ACME via Let's Encrypt). User roles: `Anonymous` (read-only) / `Registered` (read-only, logged in) / `Family` (trusted household tier, promotion-only — the audience for the upcoming `min_role`-gated Library; Phase CZ) / `Admin` (edit). The very first user to register is automatically promoted to `Admin` (enforced inline in `UserDao::create`); everyone after lands at `Registered`. The trust ladder is the explicit `Role::rank()` (Anonymous 0 < Registered 1 < Family 2 < Admin 3) — **never `derive(Ord)` on `Role`** (variants are declared alphabetically; a derived order ranks Admin below Anonymous — a `Role::iter()`-pinned test guards the ladder), and viewer-role derivations go through `AuthenticationState::role()`, not inline matches.

## Common commands

- `bacon` — default dev loop. Runs `cargo run -- data/config.json`, kill-restarts on changes in `assets/`, `styles/`, `templates/`. Press `c` for clippy.
- `cargo build` / `cargo run -- data/config.json` — direct equivalents.
- `cargo test` — runs the suite. SQLx unit tests use `#[sqlx::test(migrator = "crate::db::database_handle::MIGRATOR")]` (isolated SQLite per test). `tests/` integration tests (`server.rs`, `web.rs`) spin up the web layer in-process via `hotchkiss_io::test_support::spawn_test_server()` (no IP/DNS/ACME coordinator; plain HTTP on `127.0.0.1:0`) and hit it with `reqwest`. A debug-only `POST /test/login?role=Admin|Registered|Family` seam (`web/features/test_login.rs`, `#[cfg(debug_assertions)]` — absent from release) lets them reach role-gated routes without the WebAuthn dance.
- Browser e2e run as part of **plain `cargo test`** (no longer `#[ignore]`d — needs Google Chrome installed; `cargo test --test e2e_browser` to run just them). Pure Rust via `chromiumoxide` (CDP): drives headless Chrome against `spawn_test_server`, using a CDP **virtual authenticator** to complete the real passkey registration ceremony, then checks the `/admin/analytics` auth gate. They **serialize on a shared `E2E_LOCK` mutex** — each launches its own Chrome + WebAuthn authenticator, and running the passkey ceremonies concurrently races (blows the 20s registration deadline). **No test may be `#[ignore]`d** — a `lint_no_ignored_tests` guard test scans `src/` + `tests/` and fails if the attribute reappears (ignored tests rot silently — three e2e tests drifted out of sync with the UI that way). A test that can't always run gets a **Cargo feature gate** instead: the iOS-Simulator e2e (`tests/e2e_ios.rs`, WebKit rendering checks) is gated behind `--features ios_e2e` (boots a sim, 10–60s, needs Xcode) and run on the mini, not in the default suite.
- `cargo test <test_name>` — single test (e.g. `cargo test roundtrip`).
- `biome check --write .` — format + lint for first-party JS/CSS (`assets/scripts/` + `styles/` only, per `biome.json`; vendor untouched). Biome is a brew-installed single binary — there is NO Node toolchain in this repo (the old `npm run format` Prettier setup was removed 2026-07-08 after it reformatted the whole tree AND corrupted markdown — snake_case → `page*order`, literal `*` → `\*`; Biome doesn't format markdown at all, killing that failure class). Style-war rules (`useArrowFunction`, `useTemplate`, `useOptionalChain`, …) are off — the vanilla-IIFE defensive-`&&` browser-JS idiom is deliberate. Templates are hand-formatted (Biome HTML is still experimental; revisit when stable — Tailwind class-sorting in templates went away with Prettier).
- `./build/macos/build.sh [--profile beta|prod]` — builds an ad-hoc-signed app bundle (no notarization, no `.pkg`; the DEPLOY re-signs it with a Developer ID via the GUI-session signer agent for durable Full Disk Access — Phase CP, see the deploy section). `--profile` (default `prod`) picks the bundle name + identifier so beta and prod coexist in `/Applications` and LaunchServices: `prod` → `Hotchkiss-IO.app` / `io.hotchkiss.web`, `beta` → `Hotchkiss-IO-Beta.app` / `io.hotchkiss.web.beta`. Honors `CARGO_TARGET_DIR`; writes `$CARGO_TARGET_DIR/<name>.app` and prints `BUILT_APP=<absolute-path>` + `PROFILE=<profile>` on stdout. Resolves `VERSION` from `$VERSION` → `$GITHUB_REF_NAME` → `git describe`, falling back to `0.0.0-dev` — on the mini the post-receive extracts a `.git`-less tree, so CFBundleVersion lands on `0.0.0-dev` (cosmetic; the runtime/log/tray version comes from `CARGO_PKG_VERSION`, so bump `Cargo.toml` to match each tag — single-source fix is backlogged). The post-receive hook calls this per profile.

Deployment is push-to-deploy with an **inverted flow** (Phase 12): one bare repo at `~/repos/hotchkiss-io.git` on the Mac mini, whose `post-receive` hook (`build/macos/post-receive`) **dispatches by pushed ref** — `git push origin main` → **beta** (bleeding edge), and `git push origin vX.Y.Z` (a `v*` tag) → **prod** (deliberate promotion). For each deploying ref the hook archives the tree into a per-profile work dir, runs `build/macos/build.sh --profile <profile>`, atomically swaps the new `.app` into `/Applications`, and **gracefully restarts** the matching LaunchAgent — `launchctl kill TERM` (NOT `kickstart -k`/SIGKILL, which made macOS file a "quit unexpectedly" crash report on the mini every deploy, Phase BY); `KeepAlive=true` relaunches it onto the swapped bundle (after up to `ThrottleInterval`, kept as a crash-loop backstop), self-bootstrapping if it isn't loaded. A failed build leaves the running app untouched, and each ref deploys in an isolated subshell so one ref's failure can't skip a sibling. **Concurrency + fail-loud (Phase CO incident):** each profile's deploy holds a per-profile `mkdir` lock (macOS has no `flock`; 600s stale-break, released via an EXIT trap with the path baked in — a lazy `$local` is an unbound var under `set -u` and never releases) so OVERLAPPING pushes of the SAME profile serialize — three concurrent beta pushes once raced the `/Applications` swap, nested the `.app` into `.app.prev`, left no live bundle (beta down) while a racer still printed "deployed" + exited 0. The swap now also asserts a valid bundle (`Contents/MacOS/hotchkiss-io` executable) BEFORE the restart and ABORTS non-zero if absent — `set -e` catches an `mv` error but NOT an `mv` that nested into an existing dir (exit 0), the exact silent-corruption path. **The prod→beta media rsync carries `--max-size=2G`** so the ephemeral beta skips prod's multi-GB test assets (a 78 GB media-test file was otherwise re-copied on every beta deploy); prod + B2 keep the big files. **Releasing to prod:** bump `Cargo.toml`'s `version`, commit, `git push origin main` (lands on beta), then `git tag vX.Y.Z && git push origin vX.Y.Z`. Two LaunchAgents run side-by-side — prod `io.hotchkiss.web` on `:80`/`:443`, beta `io.hotchkiss.web.beta` on `:8080`/`:8443`. `origin` is the mini (the deploy remote); the public `github` mirror (`github.com/chotchki/hotchkiss-io`) was **re-added 2026-06-29** once the docs were de-personalized + the history rewritten to purge personal career context (Phase CM). Publishing is manual — `git push github main` + `git push github $(git tag -l 'v*')`; the mini stays the deploy path and nothing auto-pushes to GitHub. (Note: the mini's pre-CM `v*` tags still point at the OLD pre-rewrite lineage — they were deliberately NOT force-updated, since pushing 35 moved tags to the mini would trigger 35 prod deploys; GitHub carries the rewritten tags. New releases tag from the rewritten `main`, so both stay clean going forward.) **Code signing (Phase CP):** `build.sh` ad-hoc-signs (so the binary runs), then the deploy **re-signs with chris's Developer ID** — a STABLE identity so the macOS **Full Disk Access** (TCC) grant survives deploys (ad-hoc's per-build cdhash dropped FDA on every push → the external `/Volumes/MediaStorage4` media started *hanging*). `codesign` CANNOT reach the keychain from the non-GUI sshd hook (`errSecInternalComponent`; `launchctl asuser` into the GUI session needs root), so signing is delegated to a **GUI-session LaunchAgent** (`io.hotchkiss.signer` / `build/macos/sign-agent.sh`, WatchPaths a request dir): `post-receive`'s `sign_with_devid` drops the built app's path and waits for the agent's result **before the swap** — a sign failure ABORTS the deploy (never ships an ad-hoc binary that would silently drop FDA). NO hardened runtime (`--options runtime`): we don't notarize, and Dev-ID + hardened runtime enables library validation that `OS_REASON_CODESIGNING`-kills the app at launch; TCC keys on the Team ID regardless. **FDA is a ONE-TIME grant** per app (System Settings → Privacy & Security → Full Disk Access), then it sticks across every deploy. The swap also stopped `rm`ing `.app.prev` in-deploy (unlinking the exiting old process's mmap'd executable made the kernel `OS_REASON_CODESIGNING`-kill it → a "quit unexpectedly" popup, the other half of the Phase BY goal); the NEXT deploy's top-`rm` reclaims it. **Mini setup** — cert import + `security set-key-partition-list` + `build/macos/install-signer-agent.sh` (scp's the agent/hook to the mini + self-tests signing through the agent) — is in `build/macos/SETUP.md` §9. Notarization / `.pkg` stay retired (signing-IDENTITY only, for TCC stability).

**Beta instance** (`beta.hotchkiss.io:8443`) is public (grey-cloud A record that beta's own `DnsProviderService` tracks to the public IP, reached via a router `:8443` forward) and serves its **own** LE-prod cert — beta deploys as a release build, so the cert is natively trusted (the iPhone installs the PWA with no profile). Every `main` push **snapshots prod's `database.sqlite` into beta** (`snapshot_prod_db_into_beta` in the hook, via `sqlite3 .backup` of the live prod DB): it scrubs `crypto_keys` EXCEPT id 2 (beta regenerates its session signing key id 1 but KEEPS the media-URL HMAC key, so its `url_key`s match prod's), `tower_sessions`, `request_log` (visitor privacy), and `instant_acme_domains` (prod's LE account key); **1-for-1 rsyncs prod's media into beta's media** (content-addressed, `--ignore-existing` so incremental): prod's and beta's `media_paths` are MATCHING ordered roots, so `prod_roots[i]→beta_roots[i]` keeps each root on the SAME class of drive — beta's root[0] is `/Volumes/MediaStorage4/website_beta`, so the big external media stays external (consolidating prod's multi-drive media onto beta's one internal dir was the v0.0.83 boot-disk-fill, fixed in CP) + a drive-present guard skips an unmounted root; beta's config sets `media_paths`/`backup_path` under `io.hotchkiss.web.beta/`, since `app_support` is hardcoded to `io.hotchkiss.web`; **preserves** beta's own cert + ACME account (captured/restored, so beta never re-orders against the 5/week LE duplicate-cert limit); and **carries prod's `users`/passkeys** so chris's prod passkey authenticates on beta — beta's `webauthn_rp_id` is the registrable parent `hotchkiss.io` (set in beta's config), and the WebAuthn `rp_origin` includes the non-default `:8443` port. Beta data is therefore ephemeral by design: any beta-only edit is blown away on the next `main` push.

## Build-time machinery (build.rs)

`build.rs` does five things that surprise people:

1. **Creates an ephemeral SQLite schema DB at `$OUT_DIR/schema.db`** and runs `src/db/migrations/` against it, then writes `DATABASE_URL` into `.env` and exports it to rustc. This is what lets `sqlx::query!` / `query_as!` macros type-check at compile time. **If you add a migration or change a query, you may need `cargo clean -p hotchkiss-io` (or delete the OUT_DIR `schema.db`) for sqlx macros to re-validate.**
2. **Downloads a pinned Tailwind CLI** (`TAILWIND_VERSION` in `build.rs`, currently `v4.3.0`, arm64 macOS standalone binary) into `OUT_DIR` (cached under a version-keyed filename, so bumping the pin re-downloads) and compiles `styles/tailwind.css` → `assets/styles/main.css`. The compiled CSS is gitignored. (DaisyUI used to be downloaded here too but was never wired into `tailwind.css` — removed 2026-05; the site is styled with hand-rolled Tailwind utilities.) **`@source` gotcha:** Tailwind only scans the dirs listed as `@source` in `tailwind.css` — currently `templates`, `src/web/markdown`, `src/web/features`, `assets/scripts` (first-party JS builds player/upload DOM with utility classes; added in DD — before that its classes only worked by coinciding with template-used ones) — with matching `rerun-if-changed` in `build.rs`. Utility classes emitted from **Rust string builders** (the media embeds in `web/features/media.rs`, the markdown transformer) only get generated because those dirs are scanned — **a class used ONLY in a `.rs` string outside a scanned dir silently produces no CSS** (the class name has to appear as literal text somewhere Tailwind reads). Add a new emitter's dir to `@source` + `rerun-if-changed`, or the styles just won't exist at runtime.
3. **Generates the inline-SVG icon set** (Phase CN): reads the vendored Font Awesome Free 6.7.2 solid SVGs in `build/svg-icons/solid/` (kept OUTSIDE `assets/`, so `rust-embed` doesn't ship them) and emits `templates/partials/icons.html` — one parameterless askama macro per icon, called as `{% call icons::pen() %}` → inline `<svg class="icon" fill="currentColor">` (`.icon` is `height:1em;width:auto;fill:currentColor` in `tailwind.css`, so it inherits text size/color; wrap a call in a sized `<span>` for a bigger/recolored icon). **FontAwesome is GONE** — the 155 KiB `fa-solid-900.woff2` + ~18 KiB CSS were the single biggest LCP critical-path cost (PSI render-blocking ~1.9s). **To add a UI icon: drop its SVG in `build/svg-icons/solid/<name>.svg` and call `icons::<name_with_underscores>()`** — a typo'd macro is a compile error (no un-vendored icon sneaks in, so there's no template scan). WRITE-IF-CHANGED so an unchanged regen doesn't bump the mtime + cascade a recompile (the partial lands under the `templates` rerun-trigger); the generated `icons.html` is gitignored. The STL-viewer loading spinner is a self-contained `.stl-spinner` CSS ring (no icon, no dep on Tailwind scanning JS).
4. **Downloads the pinned fab-gui WASM editor bundle** (`FAB_GUI_VERSION` / `FAB_GUI_TAG` in `build.rs`, currently `0.12.0` / `web-v0.12.0`) from the `chotchki/fab-scad` GitHub release (asset `fab-gui-<version>.tar.gz`) into `$OUT_DIR/fab-gui` for `rust-embed` — the slicer/placer editor served under `/3d/editor` (Phase CW). fab-gui replaced the old fab-web/Bevy bundle (ONE codebase desktop + web; the OpenSCAD side-module is gone — scad-rs renders in the `geom/` Manifold worker now, so the bundle is `fab_gui.js` + `fab_gui_bg.wasm` + a `geom/` worker tree + a ROOT `libs.json`, no `openscad/`). Version-keyed by a sibling `fab-gui.version` marker so a bump re-downloads (same shape as the Tailwind CLI fetch), and it sha256-verifies EVERY file against the bundle's `manifest.json` (a corrupt/tampered download fails the build). Then it **drops each raw wasm ONLY when its `.gz` sibling ships** — `editor_asset` reconstructs identity by GUNZIPPING the `.gz` for a no-`Accept-Encoding` client (it can't brotli-decode), so a wasm that ships only `.br` MUST keep its raw or that client 500s (fab-web shipped the geom wasm `.br`-only; the conditional drop is safe-by-construction either way, guarded by `editor_geom_wasm_identity_for_no_encoding_client`). Exposes `FAB_GUI_VERSION` as a `rustc-env` so the runtime version-paths the editor's resource URLs (`/3d/editor/<ver>/…`) for immutable cache-busting. **The pin is a hard gate:** `cargo build` 404s loudly if `FAB_GUI_TAG` doesn't match a published release, and after bumping the pin run `cargo clean -p hotchkiss-io` (the marker short-circuit caches the last fetch). The serving half is `web/features/three_d.rs` — a COOP/COEP-scoped `/3d/editor` document (cross-origin isolation confined to that route), a flex-column frame (site header + the `#fab-gui` canvas the app binds to) with a boot splash that lifts on the app's `fab-gui:ready` event, and a `{*path}` wildcard that serves the precompressed `.br`/`.gz` variant a client advertises (with the matching `Content-Encoding`, so it skips the site CompressionLayer — never double-compressed). Covered by `tests/three_d.rs` (serving) + `tests/e2e_browser.rs::editor_boots_in_browser` (real headless-Chrome boot: subresources load, canvas present, cross-origin isolated, splash lifts, no wasm error).
5. Re-runs only when `assets/scripts`, `templates`, `migrations`, or `build/svg-icons` change.

`assets/` and `templates/` are bundled into the binary at compile time via `rust-embed` (`web/static_content.rs`) and `askama` respectively — there are no loose static files at runtime besides the SQLite database.

## Runtime architecture

`src/main.rs` → `lib::real_main` → `tray-wrapper` (macOS menu-bar anchor) → `create_server` closure → `ServiceCoordinator`.

The coordinator (`src/coordinator/service_coordinator.rs`) wires four long-lived tokio tasks connected by `broadcast` channels:

```
IpProviderService ──IPs──▶ DnsProviderService (Cloudflare API)
                                 │
AcmeProviderService ──RustlsConfig──▶ EndpointsProviderService (HTTP/HTTPS axum servers)
```

- **IpProviderService** polls `https://1.1.1.1/cdn-cgi/trace` hourly (parsing the `ip=` line) and broadcasts the public IP. Connecting to the IPv4 literal forces a v4 path. In `debug_assertions` builds it forces `127.0.0.1` so dev never touches the public DNS.
- **DnsProviderService** updates Cloudflare A records when the IP differs from DNS.
- **AcmeProviderService** loads or orders a Let's Encrypt cert (DNS-01 via Cloudflare), persists it in SQLite (`certificates` table), broadcasts a `RustlsConfig`, and refreshes every 6h. Serves **HTTP/2** (ALPN `h2`, h1.1 fallback; no HTTP/3) — the ALPN policy + the axum-server-needs-ALPN-or-it-silently-stays-h1 gotcha live in `build_server_config` and its unit test.
- **EndpointsProviderService** waits for a `RustlsConfig` then starts both an HTTP→HTTPS redirect server on `:80` and the real axum app on `:443` (served with `into_make_service_with_connect_info::<SocketAddr>()` so handlers/middleware can see the client IP). It also owns the `tower-sessions-sqlx-store` session GC task, a daily `request_log` prune task (90-day retention), and a **daily DB backup task** (`coordinator/backup.rs`): each tick `VACUUM INTO`s a dated snapshot (`database-YYYY-MM-DD.sqlite`) under `Settings::backup_path`, then prunes to a rolling 7 most-recent. The backup loop never returns and matches+logs every fallible step (`tracing::error!`) so a backup failure can't bubble out and trip the coordinator's `try_join!` (which would take the whole app down) — a bad day self-heals on the next tick. Backups exist only on disk; Backblaze covers off-site.

If any task returns, `tokio::try_join!` fails the whole coordinator — there is no auto-restart.

## Web layer (`src/web/`)

- **Routing** is composed in `web/router.rs`: `/` serves the **featured landing page** (`web/features/home.rs` → `templates/home.html`, Phase 13 — replaced the old `redirect_to_first_page`): the global `base.html` hero (photo + name + tagline) plus a content block of a one-line "what I do" + GitHub/email links, three pillar DOORS (Projects `/projects` · Blog `/blog` · Résumé `/resume` — the live, distinct destinations; Software+3D fold into `/projects` until the Phase-15 gallery, then split — it's plain template markup), and TWO content bands: **Featured** (hand-pinned) above **Latest** (auto). **Featured (Phase 13.8):** pages carrying a reserved `featured` tag in `page_category` — a **Pin button** in the page editor `POST`s `/admin/pages/{id}/feature`, which read-modify-writes the tag via `web/util/category.rs` (the column is treated as a comma-separated TAG list, so a page can be BOTH categorized and pinned — `"3d, featured"`; `ContentPageDao::set_category` deliberately does NOT stamp `modified_date`, like `set_order`, so pinning doesn't churn the feed). The field was fully plumbed but read by nothing → **no migration**. Featured is ordered by the manual **`page_order`** (the same control the /projects drag-reorder sets), recency-tiebroken (page_order is per-parent, so a blog↔project tie falls back to date). **Latest** = newest NON-featured across `blog` + `projects` (same fetch the unified feed uses, merged + sorted by creation date), so publishing freshens the front door with NO curation step; a pinned page shows in Featured, not twice. Both bands omit when empty. (`redirect_to_first_page` is retired.) `/login`, `/pages`, `/projects`, `/blog`, `/admin`, `/media` are nested routers (`/attachments` was retired in BZ.8), and `static_content()` is merged in. A custom **404** (`web/features/not_found.rs`) — the "blame the cat" page (`templates/404.html`: three cat suspects, tap-to-accuse quip overlay via a pure-Tailwind native-`<details>` `group-open:` reveal, NO JS and NO custom CSS, same no-JS approach as the nav hamburger; cat photos are web-optimized AVIF under `assets/images/404/`) renders from ONE `render_not_found` helper wired to BOTH the global router `fallback` (unmatched routes; attached **before** `with_state` so it can build the nav) AND the `get_page_path` `/pages/<slug>` miss branch (so a dead link like `/pages/Resume` lands there too — the mutating `/pages` handlers keep their plain `"No such page"`). It returns a real `404` by wrapping `HtmlTemplate` (which always renders `200`) in `(StatusCode::NOT_FOUND, _)`; a nav-build DB failure degrades to a plain 404, never a 500. `LiveReloadLayer` is added **only** in debug builds. The outer `ServiceBuilder` stack adds (outermost first) a request-logging middleware (`web/middleware/request_log.rs` → `request_log` table, fire-and-forget insert), the `TraceLayer`, the session layer, and compression. The admin `/admin/analytics` dashboard (`web/features/admin/analytics.rs`, expanded in **Phase CQ**) separates real-visitor SIGNAL from bot/scanner NOISE over a time window (7/30/90d presets OR a custom `?from=&to=` range — Phase CT), on-the-fly `GROUP BY` (no rollup table). **Two orthogonal axes that never conflate:** STATUS is factual (`count_by_status_bucket` — 2xx/3xx/403/404/4xx/5xx, 403+404 split out), AGENT is inferred (a STORED `is_bot` column, migration `0022` — **Phase CR promoted it from CQ's query-time `ua_class` VIEW (`0020`, now dropped) for speed**; the single-source classifier `request_log::is_bot(ua)` stamps it at write, an idempotent startup backfill (`coordinator/backfill_is_bot.rs`) fills legacy rows, and `POST /admin/analytics/reclassify-bots` re-runs it over history so the ruleset stays RETUNABLE despite being stored — just not re-derived on every query). The headline numbers DEFAULT to All (factual) with an always-visible **All/Humans/Bots 3-chip** (`?audience=`, threaded through the count queries as a `(?N IS NULL OR is_bot=?N)` predicate so the `ts` index still bounds the scan; a bad `?audience` degrades to All, never a 500). **The chart is a vendored-d3@7 line chart** (`assets/vendor/d3/`, admin-only + `defer`'d — off the public LCP path Phase CN protects; `assets/scripts/analytics-chart.js` ports recon-gen's `renderLineChart`): the server emits a NUMERIC-only JSON island (`shape_timeseries` → a typed serde struct, UTC zero-filled continuous daily axis, `<`-escaped) that d3 draws as a Total+Unique **overlay** (the gap = the repeat/scanner story). **Control model (CQ.7):** every range/audience/paths toggle `hx-get`s the whole page and swaps only `#analytics-content` (`hx-target`/`hx-select`/`hx-push-url` inherited by the links) so chart + tables + stats + URL always agree, `href` as the no-JS fallback — so `/admin/analytics` is now **excluded from the request-logging middleware** (it `hx-get`s itself on every toggle; mirrors the `/admin/logs` self-feed guard, and the cost is the dashboard can't time its OWN render). **Noise views:** a per-IP **Noisy IPs** leaderboard (`noisy_ips` — volume-sorted, `WHERE ip IS NOT NULL` non-negotiable, a `distinct_404 >= SCAN_DISTINCT_404_THRESHOLD` **Scanner** badge, each row linking to a `/admin/analytics/ip/{ip}` **drill-down** that derives a 404-wordlist + status mix in Rust from one query — 400 on a garbage IP, 200 empty-state, never a 500), a **never-succeeded paths** scanner-signature list (`HAVING SUM(status<400)=0`), a **Content↔All** top-pages toggle (`count_by_content_path`'s `max_status` ceiling: Content=`status<400`; All surfaces 4xx/5xx probes — static prefixes excluded), **host-grouped referrers** (`web/util/referer.rs::normalize_referer` on the `url` crate — `url::Host::{Ipv4,Ipv6}` IS the IP-literal test that strips the pollution chris flagged, grouped by registrable-ish host with `www.`/`m.`/`amp.` stripped and NO `psl` dep, category-bucketed, junk COUNTED as "N polluting referers hidden" + a direct/NULL-referer count; this replaces `count_by_referer` and fixes its latent `NOT LIKE '%hotchkiss.io%'` bug that mis-swallowed `hotchkiss.io.evil.com`), and a **Server response time** section (`duration_ms`, migration `0019`, captured in the middleware — HONESTLY labeled server-handler time, NOT client page-load/LCP; `web/util/route.rs` percentiles computed Rust-side since SQLite has none, per-route p50/p95/max via a `normalize_route` router MIRROR + a slowest-requests table). **XSS boundary:** attacker-controlled path/UA/referer strings stay in auto-escaped askama TABLES, NEVER the JSON island. **Custom date range (Phase CT):** the window is a `request_log::Window { from, to }` — concrete UTC `YYYY-MM-DD HH:MM:SS` bounds, predicate `ts >= from AND ts < to` — that ALL ~15 windowed reads take (this RETIRED the old per-query `since_days` int + upper-bound-less `datetime('now', ?)`). Added `rmcp` 2.2 + `schemars` 1.0 (~19 transitive crates). **Dogfood hardening (Phases DJ/DK, `docs/newtype-pass-design.md` + `docs/mcp-dogfood-feedback.md`):** every tool param struct is `#[serde(deny_unknown_fields)]` (`additionalProperties:false`) so a typo'd argument key is a HARD `-32602`, never a silently-wrong empty listing (DJ.5); a DUPLICATE slug under a parent is caught at `PageWrite::create_page` (a `sqlx` UNIQUE-violation downcast → `PageWriteError::DuplicateSlug{slug,parent}`) and mapped to an ACTIONABLE `-32602` "a page with slug '…' already exists under <parent>" (HTTP editor path → **409**) — the raw `UNIQUE constraint failed: content_pages…` text NEVER leaks (DK.1). The security-critical `min_role` fail-closed decode is now the ONE `MinRole` newtype (`db/dao/roles.rs` — `from_stored`/`apply_write`/`rank`/`is_visible_to`/`label`; both DAOs delegate, a `Role::iter()` test pins it), and the create-boundary empty-title guard is the `Slug` newtype (`web/util/slug.rs`, `Slug::new` = slugify-then-non-empty) (DJ.1/DJ.3). **Server identity (DK.3):** `get_info` stamps `server_info.name`/`version`/`title` (`hotchkiss-io` @ `CARGO_PKG_VERSION`) — rmcp's default `Implementation::from_build_env()` resolves at the SDK's OWN compile time, so an un-set server reports as `rmcp 2.2.0` (indistinguishable from any other rmcp server in client logs/UIs); an initialize-response test pins our name+version and asserts `rmcp` doesn't leak. The **401 carries NO `WWW-Authenticate`** by design (DK.2) — a deliberate choice for a bearer-token server (it only matters for OAuth-style resource discovery, which the DI design avoids).
- **User management (Phase CC)**: `/admin/users` (`web/features/admin/users.rs`, admin-gated by the `/admin` nest's `require_admin`) lists every user (real role badge + passkey count = `json_array_length(keys)` + live API-key count) and moves users three-way between `Registered`/`Family`/`Admin` (`POST /admin/users/{id}/role`, one "Make X" button per non-current role; `role=Anonymous` is rejected `400` — it's the not-logged-in sentinel, not an account level; Family is promotion-only, registration still lands at Registered) or deletes (`DELETE /admin/users/{id}`, which wipes the user's `api_keys` first — FK — then the row, in a tx; passkeys live in the `keys` column and go with it). **The last admin is protected** from demote/delete (no lockout) — enforced in the handlers (`409`) AND hidden in the UI; self-actions are otherwise allowed. There's still **no create-user** path: users self-register via passkey (first registrant → Admin). **Live enforcement** — a cookie session stores a SNAPSHOT of the user (`Authenticated(UserDao)`, role included), so a role change / delete wouldn't bite a live session until logout/expiry. The `refresh_session_role` middleware (`web/middleware/refresh_session_role.rs`, `from_fn_with_state`, layered INNER to `api_key_auth` / OUTER to the authz layer) re-loads the cookie session's user by id each request and injects a refreshed `SessionData` (current role) — or Anonymous if the user was deleted — so demote/delete take effect IMMEDIATELY (a Bearer key already resolves fresh, so the middleware no-ops for it). One extra cheap SQLite lookup per authenticated request.
- **Cookie hardening (Phase CC)**: the session cookie — the ONLY cookie the app sets (tower-sessions) — is explicitly `HttpOnly` + `SameSite=Lax` + `Secure` (release only; `with_secure(!debug)` keeps it off for the plain-HTTP test harness / dev). `HttpOnly` is tower-sessions' default, set explicitly so it can't silently regress; an integration test asserts `Set-Cookie` carries `HttpOnly` + `SameSite`.
- **Authorization is one fail-closed layer** (`web/middleware/require_admin_for_mutations.rs`, Phase E): **GET/HEAD/OPTIONS are public site-wide; every other method requires an authenticated Admin BY DEFAULT.** It's a default-DENY allowlist of safe methods (not a deny-list of known-mutating verbs), so a newly-added route or an exotic verb (e.g. the `/pages/preview` PATCH) is gated automatically — the secure default protects against the per-handler-omission bug it replaced (v0.0.49: an anonymous caller could mutate `/pages`). The only exceptions are an EXACT `(path, method)` allowlist of the anonymous WebAuthn ceremony POSTs (`/login/finish_authentication`, `/login/finish_register`) + the debug `/test/login` seam, and the **role-scoped mutation allowlist** (Phase CZ): exact-match `(Method, path, min Role)` entries checked by `rank()` — shipped EMPTY (a pinned test says so) until Phase DF adds `POST /library/progress` at `Family`. Its two conventions keep it a table, not a router: resource ids ride the request BODY (entries stay exact-match, never path patterns), and per-resource authz beyond the coarse role gate lives in the handler. Decision order: safe methods → WebAuthn allowlist → role-scoped allowlist → admin fallback → deny. Wired in `router.rs` **inner to the session layer** (so `SessionData` is loaded; it defaults to `Anonymous` → a clean deny, not a panic). **401 vs 403 (Phase DK.2):** the deny splits by identity — a MISSING identity (no session / no valid key → Anonymous) gets **401** (`unauthorized_response`, "Who goes there?"), an authenticated-but-INSUFFICIENT caller gets **403** (`forbidden_response`, "How about NO!"). Both live in `error_page.rs`, both HX-Redirect to `/login` for an htmx request, and the **401 deliberately carries NO `WWW-Authenticate`** — that header would pop a browser basic-auth dialog AND trigger an MCP client's OAuth discovery (the DI design avoids the chase). This is ONE site-wide rule (not `/mcp`-special): a no-key `POST /mcp` 401s exactly like an anonymous editor mutation. The `/admin` nest's `require_admin` splits identically. The **`/admin` nest keeps its own `require_admin` layer** (`web/middleware/require_admin.rs`) — it gates a _non-public_ GET (`/admin/analytics`) that the GET-public default would otherwise expose; the two are complementary. **No per-handler authz checks remain** — the layer is the single source of truth for mutations. (`list_page_attachments` GET was made public in Phase F: the attachment _content_ is already unauthenticated, so the listing leaks nothing.)
- **Errors**: handlers return `Result<Response, AppError>`. `AppError` wraps `anyhow::Error`, logs with a UUID trace id, and returns a 500 with that id to the client.
- **`AppError` swallows status info** — if a handler needs to return e.g. a 404 or 403, return `Ok((StatusCode::X, "msg").into_response())` instead of bubbling via `?`. This is a deliberate pattern, see existing handlers for examples.

## Content model

It REUSES the editable post-date, so NO new column and NO cron: the flip is evaluated at read time. Single source of truth: `ContentPageDao::is_scheduled()` (`page_creation_date > Utc::now()`, strictly `>` so exactly-now is live) + `is_visible_to` (special pages exempt from the SCHEDULING clause; since Phase DA the fn takes `viewer: Role` and adds the role clause — see the DA block below for the current formula). Enforcement is HYBRID by necessity — **Rust-side** `is_visible_to()` on the decoded `DateTime<Utc>` for the direct-serve / feed / home / nav paths (sidesteps the column's mixed TEXT date formats), and **SQL-side** `datetime(page_creation_date) <= datetime('now')` threaded into the paginated `count_children` / `find_children_{newest,ordered}_paged` (an admin-exemption bind derived from the `viewer: Role` param): the COUNT and the ROW fetch MUST share the identical predicate or the pager desyncs, and it MUST `datetime()`-normalize (the column stores both the space form from `CURRENT_TIMESTAMP` and the `T+00:00` RFC3339 form sqlx writes on any edit, so a raw string `>` compares them wrong). `find_by_path` stays DATE-BLIND — it's shared with the put/delete/post mutations, so the gate lives in the READ handlers, never the DAO. **Direct-serve is security-critical:** `get_page_path` scans the WHOLE resolved `pages_path` (`.iter().all(is_visible_to)`), so a future-dated non-special ANCESTOR hides its entire subtree (a leaf-only gate would leak the parent's title in the breadcrumb), and both it and `show_post` return the SAME `render_not_found` cat-404 for a scheduled hit as for a genuine miss — a different response is an existence ORACLE. That oracle rule also re-gated **`redirect_to_first_page`** (`/pages` now redirects to the first VISIBLE top-level page, not blindly `.first()` — a scheduled first page was leaking its slug in the `Location` header; caught in review). `show_post` filters future siblings out of the prev/next nav; `/resume`'s `newest_resume_child` dropped its `LIMIT 1` to pick the newest VISIBLE child (a scheduled draft no longer 404s the whole route), and `show_resume_pdf` gained the `SessionData` it lacked (it was serving drafts with zero auth context). Fenced code keeps its `language-*` class and **highlight.js** (`assets/vendor/highlightjs/`, init `assets/scripts/code-highlight.js`) highlights `pre:not(.d2-source) code` (the d2 source is excluded). Both load in `get_page.html` and degrade to readable source without JS. **(Phase BW)** the BFS walk descends into ALL containers (lists, headings, blockquotes, emphasis, …), so math/images/diagrams nested anywhere convert — not just at top level (a Root/Paragraph-only walk was a BV regression that left list-nested math as raw `$…$`). **GFM tables** are enabled (`gfm_table`); since `mdast_util_to_markdown` (0.0.1) can't re-serialize a `Table` node, each table is rendered from its original markdown source slice in the walk and emitted as HTML (bypassing the AST round-trip that would otherwise drop it; markdown-rs `Point.offset` is a BYTE offset, so the slice is char-safe). **Panic-hardened (Phase CG):** the alpha `markdown-rs` / `mdast_util_to_markdown` can PANIC on edge-case content — a Wayback-recovered 2012 post (smart quotes + escaped angle brackets) did, and since an uncaught handler panic RESETS the connection (a `000`, no 500) AND the feed transforms EVERY post, that one post took down its page *and* the whole feed. Two layers fix it: `transform()` wraps the work in `std::panic::catch_unwind` and degrades to the escaped source on panic (so a page/feed always render something safe), and a router-level `CatchPanicLayer` (`router::handle_panic`) turns ANY handler panic into a styled 500 instead of a dropped connection. **(Phase CQ.1.1)** the request-logging middleware is layered OUTER to `CatchPanicLayer` — so a caught panic-500 flows back out through the log layer and IS recorded in `request_log` (before the reorder the log insert sat INSIDE the catch and the panic unwound past it, so panic-500s were invisible to the access log + analytics). **Render caching (Phase CS):** `transform()` and `excerpt()` are deterministic pure functions of their input markdown, so EVERY page + feed handler now calls them through `web/markdown/render_cache.rs` (`cached_transform` / `cached_excerpt`) — a process-lifetime `LazyLock<Mutex<HashMap>>` keyed by the 128-bit content hash, same pattern as the diagram/résumé-PDF caches. Content-hash keying makes invalidation FREE: an edit changes the markdown → a new key → a miss → a fresh render, and the stale entry just goes cold — NO write-path wiring. It MUST stay in-memory (never persisted): `transform()`'s one side effect is registering ` ```d2 ` sources into the process-lifetime diagram `REGISTRY`, and a cache HIT skips that call — coherent ONLY because the cache and the registry share process lifetime (a hit implies `transform` already ran this process, so the registry is populated; a disk-persisted cache would survive a restart that empties the registry → `/diagram/<hash>` 404s). The `/pages/preview` live-preview path deliberately stays UNCACHED (ephemeral unsaved content would pollute the cache with one-off entries).

## Configuration

`Settings` is loaded from JSON, in this order:

1. First CLI arg, if present (this is what `bacon` and dev use: `data/config.json`).
2. Otherwise `~/Library/Application Support/io.hotchkiss.web/config.json` on macOS, `$HOME/...` elsewhere.

Required fields: `cloudflare_token`, `domain`. Optional path fields — `database_path`, `log_path`, `cache_path`, `backup_path` — default to `~/Library/Application Support/io.hotchkiss.web/data/database.sqlite`, `~/Library/Logs/io.hotchkiss.web`, `~/Library/Caches/io.hotchkiss.web`, and `~/Library/Application Support/io.hotchkiss.web/backups` respectively when omitted (see `Settings::resolve` in `src/settings.rs`). `backup_path` is where the daily DB snapshots land (created on first backup if missing). The on-disk JSON is deserialized into a private `RawSettings`; unknown keys are silently ignored. The `data/` directory in the repo is gitignored and contains the dev config + SQLite database.

The macOS app is **not sandboxed** (the `com.apple.security.app-sandbox` entitlement was dropped, 2026-05) — `NSHomeDirectory()` returns the real home, so the paths above are literal. On the production Mac the app runs as a LaunchAgent (`build/macos/io.hotchkiss.web.plist`) in the user GUI session, which keeps the tray icon alive; macOS Mojave+ lets the non-root process bind ports 80/443 because axum binds `INADDR_ANY`.

## Things to watch out for

- **Vendored OpenSSL**: `openssl` is pinned with `features = ["vendored"]` to avoid system-OpenSSL packaging issues.
- **Ports default to 80/443 but are configurable** (`http_port` / `https_port` in `Settings`, since Phase 11; beta runs 8080/8443) — running locally as non-root on 80/443 will fail to bind, so set high ports in your dev config or accept the bind failure. (On the production Mac 80/443 work unprivileged because macOS Mojave+ allows non-root binds to ports <1024 when binding `INADDR_ANY`.)
- **`assets/styles/main.css` is generated** — never edit it; edit `styles/tailwind.css` and let `build.rs` recompile.
- **Pushing deploys (inverted flow, Phase 12)** — `git push origin main` rebuilds + restarts **beta**; only a `v*` **tag** push rebuilds + restarts **prod** (~10–15s downtime; the restart is a graceful `launchctl kill TERM` + `KeepAlive` relaunch — Phase BY — not a SIGKILL). Prod no longer auto-updates from `main`. A failed build aborts before the swap (the running app keeps serving), but a *successful build of broken runtime behavior* goes live on whichever target the ref maps to. `origin` is the mini (deploy); `github` is the public mirror, re-added 2026-06-29 after the Phase CM docs-scrub + history rewrite — push `main`/`v*` tags there manually to publish (nothing auto-pushes).
- CI layout: `test_and_coverage.yml` runs on `macos-latest` per push (lint + test + grcov coverage), `check_ip.yml` is a small scheduled IP check — both are informational and gate nothing. The `release.yml` / `install.yml` GitHub Actions workflows were deleted (2026-05); deployment is the `git push origin main` → `post-receive` hook flow described under "Common commands". Both Actions went dormant 2026-06 (no GitHub remote); with the mirror re-added 2026-06-29 (Phase CM) they run again on each manual `git push github` — still informational, gating nothing.
