# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Christopher Hotchkiss's personal site / CRM. A single self-contained Rust binary that ships as a macOS `.app` with a tray icon, serves the site over HTTPS, and self-manages its own DNS (Cloudflare) and TLS certificates (ACME via Let's Encrypt). User roles: `Anonymous` (read-only) / `Registered` (read-only, logged in) / `Family` (trusted household tier, promotion-only ‚Äî the audience for the upcoming `min_role`-gated Library; Phase CZ) / `Admin` (edit). The very first user to register is automatically promoted to `Admin` (enforced inline in `UserDao::create`); everyone after lands at `Registered`. The trust ladder is the explicit `Role::rank()` (Anonymous 0 < Registered 1 < Family 2 < Admin 3) ‚Äî **never `derive(Ord)` on `Role`** (variants are declared alphabetically; a derived order ranks Admin below Anonymous ‚Äî a `Role::iter()`-pinned test guards the ladder), and viewer-role derivations go through `AuthenticationState::role()`, not inline matches.

## Common commands

- `bacon` ‚Äî default dev loop. Runs `cargo run -- data/config.json`, kill-restarts on changes in `assets/`, `styles/`, `templates/`. Press `c` for clippy.
- `cargo build` / `cargo run -- data/config.json` ‚Äî direct equivalents.
- `cargo test` ‚Äî runs the suite. SQLx unit tests use `#[sqlx::test(migrator = "crate::db::database_handle::MIGRATOR")]` (isolated SQLite per test). `tests/` integration tests (`server.rs`, `web.rs`) spin up the web layer in-process via `hotchkiss_io::test_support::spawn_test_server()` (no IP/DNS/ACME coordinator; plain HTTP on `127.0.0.1:0`) and hit it with `reqwest`. A debug-only `POST /test/login?role=Admin|Registered|Family` seam (`web/features/test_login.rs`, `#[cfg(debug_assertions)]` ‚Äî absent from release) lets them reach role-gated routes without the WebAuthn dance.
- Browser e2e run as part of **plain `cargo test`** (no longer `#[ignore]`d ‚Äî needs Google Chrome installed; `cargo test --test e2e_browser` to run just them). Pure Rust via `chromiumoxide` (CDP): drives headless Chrome against `spawn_test_server`, using a CDP **virtual authenticator** to complete the real passkey registration ceremony, then checks the `/admin/analytics` auth gate. They **serialize on a shared `E2E_LOCK` mutex** ‚Äî each launches its own Chrome + WebAuthn authenticator, and running the passkey ceremonies concurrently races (blows the 20s registration deadline). **No test may be `#[ignore]`d** ‚Äî a `lint_no_ignored_tests` guard test scans `src/` + `tests/` and fails if the attribute reappears (ignored tests rot silently ‚Äî three e2e tests drifted out of sync with the UI that way). A test that can't always run gets a **Cargo feature gate** instead: the iOS-Simulator e2e (`tests/e2e_ios.rs`, WebKit rendering checks) is gated behind `--features ios_e2e` (boots a sim, 10‚Äì60s, needs Xcode) and run on the mini, not in the default suite.

- `biome check --write .` ‚Äî format + lint for first-party JS/CSS (`assets/scripts/` + `styles/` only, per `biome.json`; vendor untouched). Honors `CARGO_TARGET_DIR`; writes `$CARGO_TARGET_DIR/<name>.app` and prints `BUILT_APP=<absolute-path>` + `PROFILE=<profile>` on stdout. Resolves `VERSION` from `$VERSION` ‚Üí `$GITHUB_REF_NAME` ‚Üí `git describe`, falling back to `0.0.0-dev` ‚Äî on the mini the post-receive extracts a `.git`-less tree, so CFBundleVersion lands on `0.0.0-dev` (cosmetic; the runtime/log/tray version comes from `CARGO_PKG_VERSION`, so bump `Cargo.toml` to match each tag ‚Äî single-source fix is backlogged). The post-receive hook calls this per profile.

Deployment is push-to-deploy with an **inverted flow** (Phase 12): one bare repo at `~/repos/hotchkiss-io.git` on the Mac mini, whose `post-receive` hook (`build/macos/post-receive`) **dispatches by pushed ref** ‚Äî `git push origin main` ‚Üí **beta** (bleeding edge), and `git push origin vX.Y.Z` (a `v*` tag) ‚Üí **prod** (deliberate promotion). For each deploying ref the hook archives the tree into a per-profile work dir, runs `build/macos/build.sh --profile <profile>`, atomically swaps the new `.app` into `/Applications`, and **gracefully restarts** the matching LaunchAgent ‚Äî `launchctl kill TERM` (NOT `kickstart -k`/SIGKILL, which made macOS file a "quit unexpectedly" crash report on the mini every deploy, Phase BY); `KeepAlive=true` relaunches it onto the swapped bundle (after up to `ThrottleInterval`, kept as a crash-loop backstop), self-bootstrapping if it isn't loaded. A failed build leaves the running app untouched, and each ref deploys in an isolated subshell so one ref's failure can't skip a sibling. **Concurrency + fail-loud (Phase CO incident):** each profile's deploy holds a per-profile `mkdir` lock (macOS has no `flock`; 600s stale-break, released via an EXIT trap with the path baked in ‚Äî a lazy `$local` is an unbound var under `set -u` and never releases) so OVERLAPPING pushes of the SAME profile serialize ‚Äî three concurrent beta pushes once raced the `/Applications` swap, nested the `.app` into `.app.prev`, left no live bundle (beta down) while a racer still printed "deployed" + exited 0. The swap now also asserts a valid bundle (`Contents/MacOS/hotchkiss-io` executable) BEFORE the restart and ABORTS non-zero if absent ‚Äî `set -e` catches an `mv` error but NOT an `mv` that nested into an existing dir (exit 0), the exact silent-corruption path. **The prod‚Üíbeta media rsync carries `--max-size=2G`** so the ephemeral beta skips prod's multi-GB test assets (a 78 GB media-test file was otherwise re-copied on every beta deploy); prod + B2 keep the big files. **Releasing to prod:** bump `Cargo.toml`'s `version`, commit, `git push origin main` (lands on beta), then `git tag vX.Y.Z && git push origin vX.Y.Z`. Two LaunchAgents run side-by-side ‚Äî prod `io.hotchkiss.web` on `:80`/`:443`, beta `io.hotchkiss.web.beta` on `:8080`/`:8443`. `origin` is the mini (the deploy remote); the public `github` mirror (`github.com/chotchki/hotchkiss-io`) was **re-added 2026-06-29** once the docs were de-personalized + the history rewritten to purge personal career context (Phase CM). Publishing is manual ‚Äî `git push github main` + `git push github $(git tag -l 'v*')`; the mini stays the deploy path and nothing auto-pushes to GitHub. (Note: the mini's pre-CM `v*` tags still point at the OLD pre-rewrite lineage ‚Äî they were deliberately NOT force-updated, since pushing 35 moved tags to the mini would trigger 35 prod deploys; GitHub carries the rewritten tags. New releases tag from the rewritten `main`, so both stay clean going forward.) **Code signing (Phase CP):** `build.sh` ad-hoc-signs (so the binary runs), then the deploy **re-signs with chris's Developer ID** ‚Äî a STABLE identity so the macOS **Full Disk Access** (TCC) grant survives deploys (ad-hoc's per-build cdhash dropped FDA on every push ‚Üí the external `/Volumes/MediaStorage4` media started *hanging*). `codesign` CANNOT reach the keychain from the non-GUI sshd hook (`errSecInternalComponent`; `launchctl asuser` into the GUI session needs root), so signing is delegated to a **GUI-session LaunchAgent** (`io.hotchkiss.signer` / `build/macos/sign-agent.sh`, WatchPaths a request dir): `post-receive`'s `sign_with_devid` drops the built app's path and waits for the agent's result **before the swap** ‚Äî a sign failure ABORTS the deploy (never ships an ad-hoc binary that would silently drop FDA). NO hardened runtime (`--options runtime`): we don't notarize, and Dev-ID + hardened runtime enables library validation that `OS_REASON_CODESIGNING`-kills the app at launch; TCC keys on the Team ID regardless. **FDA is a ONE-TIME grant** per app (System Settings ‚Üí Privacy & Security ‚Üí Full Disk Access), then it sticks across every deploy. The swap also stopped `rm`ing `.app.prev` in-deploy (unlinking the exiting old process's mmap'd executable made the kernel `OS_REASON_CODESIGNING`-kill it ‚Üí a "quit unexpectedly" popup, the other half of the Phase BY goal); the NEXT deploy's top-`rm` reclaims it. **Mini setup** ‚Äî cert import + `security set-key-partition-list` + `build/macos/install-signer-agent.sh` (scp's the agent/hook to the mini + self-tests signing through the agent) ‚Äî is in `build/macos/SETUP.md` ¬ß9. Notarization / `.pkg` stay retired (signing-IDENTITY only, for TCC stability).

**Beta instance** (`beta.hotchkiss.io:8443`) is public (grey-cloud A record that beta's own `DnsProviderService` tracks to the public IP, reached via a router `:8443` forward) and serves its **own** LE-prod cert ‚Äî beta deploys as a release build, so the cert is natively trusted (the iPhone installs the PWA with no profile). Every `main` push **snapshots prod's `database.sqlite` into beta** (`snapshot_prod_db_into_beta` in the hook, via `sqlite3 .backup` of the live prod DB): it scrubs `crypto_keys` EXCEPT id 2 (beta regenerates its session signing key id 1 but KEEPS the media-URL HMAC key, so its `url_key`s match prod's), `tower_sessions`, `request_log` (visitor privacy), and `instant_acme_domains` (prod's LE account key); **1-for-1 rsyncs prod's media into beta's media** (content-addressed, `--ignore-existing` so incremental): prod's and beta's `media_paths` are MATCHING ordered roots, so `prod_roots[i]‚Üíbeta_roots[i]` keeps each root on the SAME class of drive ‚Äî beta's root[0] is `/Volumes/MediaStorage4/website_beta`, so the big external media stays external (consolidating prod's multi-drive media onto beta's one internal dir was the v0.0.83 boot-disk-fill, fixed in CP) + a drive-present guard skips an unmounted root; beta's config sets `media_paths`/`backup_path` under `io.hotchkiss.web.beta/`, since `app_support` is hardcoded to `io.hotchkiss.web`; **preserves** beta's own cert + ACME account (captured/restored, so beta never re-orders against the 5/week LE duplicate-cert limit); and **carries prod's `users`/passkeys** so chris's prod passkey authenticates on beta ‚Äî beta's `webauthn_rp_id` is the registrable parent `hotchkiss.io` (set in beta's config), and the WebAuthn `rp_origin` includes the non-default `:8443` port. Beta data is therefore ephemeral by design: any beta-only edit is blown away on the next `main` push.

## Build-time machinery (build.rs)

`build.rs` does four things that surprise people:

1. **Creates an ephemeral SQLite schema DB at `$OUT_DIR/schema.db`** and runs `src/db/migrations/` against it, then writes `DATABASE_URL` into `.env` and exports it to rustc. This is what lets `sqlx::query!` / `query_as!` macros type-check at compile time. **If you add a migration or change a query, you may need `cargo clean -p hotchkiss-io` (or delete the OUT_DIR `schema.db`) for sqlx macros to re-validate.**
2. **Downloads a pinned Tailwind CLI** (`TAILWIND_VERSION` in `build.rs`, currently `v4.3.0`, arm64 macOS standalone binary) into `OUT_DIR` (cached under a version-keyed filename, so bumping the pin re-downloads) and compiles `styles/tailwind.css` ‚Üí `assets/styles/main.css`. The compiled CSS is gitignored. (DaisyUI used to be downloaded here too but was never wired into `tailwind.css` ‚Äî removed 2026-05; the site is styled with hand-rolled Tailwind utilities.) **`@source` gotcha:** Tailwind only scans the dirs listed as `@source` in `tailwind.css` ‚Äî currently `templates`, `src/web/markdown`, `src/web/features` (with matching `rerun-if-changed` in `build.rs`). Utility classes emitted from **Rust string builders** (the media embeds in `web/features/media.rs`, the markdown transformer) only get generated because those dirs are scanned ‚Äî **a class used ONLY in a `.rs` string outside a scanned dir silently produces no CSS** (the class name has to appear as literal text somewhere Tailwind reads). Add a new emitter's dir to `@source` + `rerun-if-changed`, or the styles just won't exist at runtime.
3. **Generates the inline-SVG icon set** (Phase CN): reads the vendored Font Awesome Free 6.7.2 solid SVGs in `build/svg-icons/solid/` (kept OUTSIDE `assets/`, so `rust-embed` doesn't ship them) and emits `templates/partials/icons.html` ‚Äî one parameterless askama macro per icon, called as `{% call icons::pen() %}` ‚Üí inline `<svg class="icon" fill="currentColor">` (`.icon` is `height:1em;width:auto;fill:currentColor` in `tailwind.css`, so it inherits text size/color; wrap a call in a sized `<span>` for a bigger/recolored icon). **FontAwesome is GONE** ‚Äî the 155 KiB `fa-solid-900.woff2` + ~18 KiB CSS were the single biggest LCP critical-path cost (PSI render-blocking ~1.9s). **To add a UI icon: drop its SVG in `build/svg-icons/solid/<name>.svg` and call `icons::<name_with_underscores>()`** ‚Äî a typo'd macro is a compile error (no un-vendored icon sneaks in, so there's no template scan). WRITE-IF-CHANGED so an unchanged regen doesn't bump the mtime + cascade a recompile (the partial lands under the `templates` rerun-trigger); the generated `icons.html` is gitignored. The STL-viewer loading spinner is a self-contained `.stl-spinner` CSS ring (no icon, no dep on Tailwind scanning JS).
4. Re-runs only when `assets/scripts`, `templates`, `migrations`, or `build/svg-icons` change.

`assets/` and `templates/` are bundled into the binary at compile time via `rust-embed` (`web/static_content.rs`) and `askama` respectively ‚Äî there are no loose static files at runtime besides the SQLite database.

## Runtime architecture

`src/main.rs` ‚Üí `lib::real_main` ‚Üí `tray-wrapper` (macOS menu-bar anchor) ‚Üí `create_server` closure ‚Üí `ServiceCoordinator`.

The coordinator (`src/coordinator/service_coordinator.rs`) wires four long-lived tokio tasks connected by `broadcast` channels:

```
IpProviderService ‚îÄ‚îÄIPs‚îÄ‚îÄ‚ñ∂ DnsProviderService (Cloudflare API)
                                 ‚îÇ
AcmeProviderService ‚îÄ‚îÄRustlsConfig‚îÄ‚îÄ‚ñ∂ EndpointsProviderService (HTTP/HTTPS axum servers)
```

- **IpProviderService** polls `https://1.1.1.1/cdn-cgi/trace` hourly (parsing the `ip=` line) and broadcasts the public IP. Connecting to the IPv4 literal forces a v4 path. In `debug_assertions` builds it forces `127.0.0.1` so dev never touches the public DNS.
- **DnsProviderService** updates Cloudflare A records when the IP differs from DNS.
- **AcmeProviderService** loads or orders a Let's Encrypt cert (DNS-01 via Cloudflare), persists it in SQLite (`certificates` table), broadcasts a `RustlsConfig`, and refreshes every 6h. Serves **HTTP/2** (ALPN `h2`, h1.1 fallback; no HTTP/3) ‚Äî the ALPN policy + the axum-server-needs-ALPN-or-it-silently-stays-h1 gotcha live in `build_server_config` and its unit test.
- **EndpointsProviderService** waits for a `RustlsConfig` then starts both an HTTP‚ÜíHTTPS redirect server on `:80` and the real axum app on `:443` (served with `into_make_service_with_connect_info::<SocketAddr>()` so handlers/middleware can see the client IP). It also owns the `tower-sessions-sqlx-store` session GC task, a daily `request_log` prune task (90-day retention), and a **daily DB backup task** (`coordinator/backup.rs`): each tick `VACUUM INTO`s a dated snapshot (`database-YYYY-MM-DD.sqlite`) under `Settings::backup_path`, then prunes to a rolling 7 most-recent. The backup loop never returns and matches+logs every fallible step (`tracing::error!`) so a backup failure can't bubble out and trip the coordinator's `try_join!` (which would take the whole app down) ‚Äî a bad day self-heals on the next tick. Backups exist only on disk; Backblaze covers off-site.

If any task returns, `tokio::try_join!` fails the whole coordinator ‚Äî there is no auto-restart.

It REUSES the editable post-date, so NO new column and NO cron: the flip is evaluated at read time. Single source of truth: `ContentPageDao::is_scheduled()` (`page_creation_date > Utc::now()`, strictly `>` so exactly-now is live) + `is_visible_to` (special pages exempt from the SCHEDULING clause; since Phase DA the fn takes `viewer: Role` and adds the role clause ‚Äî see the DA block below for the current formula). Since `/media/file/<url_key>` has no extension, the render emits **`data-format` (stl|3mf)** on the object and the viewer branches loader on it (keeping the 3MF's own materials; only STL gets the yellow default). **Selection (by TYPE then SIZE):** for the common set of image + low-res 3MF + high-res 3MF, the **viewer = the SMALLEST `model/3mf`** (color + fast; falls back to the smallest mesh if there's no 3MF), the **download = the LARGEST `model/*`** (full-res), and the **image = the thumbnail** (library card / cover). `bytes` is the fidelity key (decimation is monotonic). `.3mf` is typed by `probe` as kind **`Stl` (the viewable-mesh kind covers 3MF)** with mime `model/3mf` ‚Äî so a model published as a 3MF-only item (bowtie's shape) renders a COLOR VIEWER, and a 3MF grouped alongside an STL LOD also views (3MF) + downloads (STL). Bambu PLATES stay downloads NOT via kind but because `fab` lists them as plain markdown LINKS ‚Äî only an `![](/media/<ref>)` embed hits the kind dispatch, and there the intent is always "show it". **Model + render image in one item:** an item can carry a mesh (STL/3MF) AND a render IMAGE (its library card thumbnail ‚Äî `show_media_library`'s `thumb_url_key` picks the image variant; a page cover can too). Two rules make this robust: (1) upload sets the item kind to the **DOMINANT** kind across grouped files (`dominant_kind`: a model or video BEATS an image), NOT the first file's ‚Äî so grouping a render with a model keeps it a viewer regardless of upload order (it used to be first-file, so an image-first group silently became an Image with no viewer); (2) the STL render selects the viewer + download over **`model/*` variants ONLY** ‚Äî an image variant is the thumbnail/poster, NEVER mis-loaded as the mesh. The site does NOT decimate ‚Äî `fab publish` provides the low-res mesh (like it provides encodes), uploaded as a grouped item or via `add_encode` (STL variants carry no `width`, so `bytes` is the fidelity key; `variant_id` fetch order is insertion order, so the render's `sort_by_key(bytes)` is load-bearing). **`fab publish` contract:** upload the full + decimated STL as ONE item (grouped drop or item+encode); ship 3MF / OpenSCAD source as SEPARATE `MediaKind::File` items (each its own download button, differentiated by `title`). **Responsive images (Phase CN):** an image upload ALSO generates width-stepped AVIF downscales (`src/media/resize.rs`, widths 480/960, skipping any ‚â• source) ‚Äî each a content-addressed `image/avif` `media_variant` carrying its pixel `width` (migration `0018` added `media_variant.width`/`height`; `media.width/height` is the original ITEM's dims, this is per-ENCODING). `render_embed_html` then emits `<img src srcset="‚Ä¶480w,‚Ä¶960w,‚Ä¶<orig>w" sizes="(max-width:768px) 100vw,768px">` (src = the largest, for a no-srcset client + the zoom view); `cover_url_for` serves the SMALLEST width-stepped variant for a ~300px card thumbnail. A legacy image with no widths degrades to a single `<img src>`. A **background, idempotent startup backfill** (`coordinator/backfill_responsive_images.rs`, spawned DETACHED ‚Äî never in the `try_join!`, so it can't delay boot or take the app down; backup-first, per-item non-fatal) generates the downscales + stamps the original's width for images uploaded before CN. The resize reuses poster.rs's in-process `image`-crate AVIF path (no new ffmpeg-build dep); rav1e is slow, so the ladder is capped at 960 + uploads run it best-effort under `spawn_blocking`. **Video sources are ordered HEVC-before-AV1** (`codec_rank`) ‚Äî Safari's `<video>` AV1 path is jerky even on AV1-HW Macs; HEVC hardware-decodes smooth (AV1 stays as the non-Apple fallback). The **library** (`web/features/admin/media.rs`, `templates/admin/media.html`, `media-upload.js`): drag-drop grouped upload (drop a video's encodes together ‚Üí one item), per-card add-encode / rename / per-stream delete / inline play / copy-ref / name search. **Inline editor upload** (`editor-support.js`): drop files on the markdown box (or the üéû toolbar button) ‚Üí async POST `/admin/media/upload` ‚Üí insert `![](/media/<ref>)` at the cursor with NO page refresh (the old attachment upload `htmx_refresh()`'d and ate unsaved edits). **Covers are media:** `content_pages.page_cover_media_id` (migration `0014`); blog/project cards render via `media::cover_url_for(page_id)` (the SMALLEST width-stepped image variant ‚Äî a video's poster; Phase CN); the editor cover field takes a media ref (`cover_ref_for` pre-fills, `put_page` resolves ‚Üí media_id). **Cover hero (Phase CV):** the DETAIL view (`get_page.html`) also renders the cover as a stacked hero banner at the top of the reader view ‚Äî `media::cover_hero_for(page_id)` picks the LARGEST width-stepped image variant (contrast `cover_url_for`'s smallest thumbnail) + a `srcset`, carried on `GetPageTemplate.hero` (populated by `get_page_path` + `show_post`; `None` on the r√©sum√©), rendered `!edit`-gated as a capped-height `object-cover` band (`data-hero`) with the page title ‚Äî plus its date + a "Scheduled" badge ‚Äî OVER the image on a navy‚Üítransparent bottom scrim, so the plain `<h1>` block is skipped whenever a hero renders (the hero's `<h1>` is the accessible title; the `<img>` is `alt=""`). ## Configuration

`Settings` is loaded from JSON, in this order:

1. First CLI arg, if present (this is what `bacon` and dev use: `data/config.json`).
2. Otherwise `~/Library/Application Support/io.hotchkiss.web/config.json` on macOS, `$HOME/...` elsewhere.

Required fields: `cloudflare_token`, `domain`. Optional path fields ‚Äî `database_path`, `log_path`, `cache_path`, `backup_path` ‚Äî default to `~/Library/Application Support/io.hotchkiss.web/data/database.sqlite`, `~/Library/Logs/io.hotchkiss.web`, `~/Library/Caches/io.hotchkiss.web`, and `~/Library/Application Support/io.hotchkiss.web/backups` respectively when omitted (see `Settings::resolve` in `src/settings.rs`). `backup_path` is where the daily DB snapshots land (created on first backup if missing). The on-disk JSON is deserialized into a private `RawSettings`; unknown keys are silently ignored. The `data/` directory in the repo is gitignored and contains the dev config + SQLite database.

The macOS app is **not sandboxed** (the `com.apple.security.app-sandbox` entitlement was dropped, 2026-05) ‚Äî `NSHomeDirectory()` returns the real home, so the paths above are literal. On the production Mac the app runs as a LaunchAgent (`build/macos/io.hotchkiss.web.plist`) in the user GUI session, which keeps the tray icon alive; macOS Mojave+ lets the non-root process bind ports 80/443 because axum binds `INADDR_ANY`.

## Things to watch out for

- **Vendored OpenSSL**: `openssl` is pinned with `features = ["vendored"]` to avoid system-OpenSSL packaging issues.
- **Ports default to 80/443 but are configurable** (`http_port` / `https_port` in `Settings`, since Phase 11; beta runs 8080/8443) ‚Äî running locally as non-root on 80/443 will fail to bind, so set high ports in your dev config or accept the bind failure. (On the production Mac 80/443 work unprivileged because macOS Mojave+ allows non-root binds to ports <1024 when binding `INADDR_ANY`.)
- **`assets/styles/main.css` is generated** ‚Äî never edit it; edit `styles/tailwind.css` and let `build.rs` recompile.
- **Pushing deploys (inverted flow, Phase 12)** ‚Äî `git push origin main` rebuilds + restarts **beta**; only a `v*` **tag** push rebuilds + restarts **prod** (~10‚Äì15s downtime; the restart is a graceful `launchctl kill TERM` + `KeepAlive` relaunch ‚Äî Phase BY ‚Äî not a SIGKILL). Prod no longer auto-updates from `main`. A failed build aborts before the swap (the running app keeps serving), but a *successful build of broken runtime behavior* goes live on whichever target the ref maps to. `origin` is the mini (deploy); `github` is the public mirror, re-added 2026-06-29 after the Phase CM docs-scrub + history rewrite ‚Äî push `main`/`v*` tags there manually to publish (nothing auto-pushes).
- CI layout: `test_and_coverage.yml` runs on `macos-latest` per push (lint + test + grcov coverage), `check_ip.yml` is a small scheduled IP check ‚Äî both are informational and gate nothing. The `release.yml` / `install.yml` GitHub Actions workflows were deleted (2026-05); deployment is the `git push origin main` ‚Üí `post-receive` hook flow described under "Common commands". Both Actions went dormant 2026-06 (no GitHub remote); with the mirror re-added 2026-06-29 (Phase CM) they run again on each manual `git push github` ‚Äî still informational, gating nothing.
