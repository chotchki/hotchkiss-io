# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Christopher Hotchkiss's personal site / CRM. A single self-contained Rust binary that ships as a macOS `.app` with a tray icon, serves the site over HTTPS, and self-manages its own DNS (Cloudflare) and TLS certificates (ACME via Let's Encrypt). User roles: `Anonymous` (read-only) / `Registered` (read-only, logged in) / `Admin` (edit). The very first user to register is automatically promoted to `Admin` (enforced inline in `UserDao::create`).

## Common commands

- `bacon` — default dev loop. Runs `cargo run -- data/config.json`, kill-restarts on changes in `assets/`, `styles/`, `templates/`. Press `c` for clippy.
- `cargo build` / `cargo run -- data/config.json` — direct equivalents.
- `cargo test` — runs the suite. SQLx unit tests use `#[sqlx::test(migrator = "crate::db::database_handle::MIGRATOR")]` (isolated SQLite per test). `tests/` integration tests (`server.rs`, `web.rs`) spin up the web layer in-process via `hotchkiss_io::test_support::spawn_test_server()` (no IP/DNS/ACME coordinator; plain HTTP on `127.0.0.1:0`) and hit it with `reqwest`. A debug-only `POST /test/login?role=Admin|Registered` seam (`web/features/test_login.rs`, `#[cfg(debug_assertions)]` — absent from release) lets them reach role-gated routes without the WebAuthn dance.
- Browser e2e run as part of **plain `cargo test`** (no longer `#[ignore]`d — needs Google Chrome installed; `cargo test --test e2e_browser` to run just them). Pure Rust via `chromiumoxide` (CDP): drives headless Chrome against `spawn_test_server`, using a CDP **virtual authenticator** to complete the real passkey registration ceremony, then checks the `/admin/analytics` auth gate. They **serialize on a shared `E2E_LOCK` mutex** — each launches its own Chrome + WebAuthn authenticator, and running the passkey ceremonies concurrently races (blows the 20s registration deadline). **No test may be `#[ignore]`d** — a `lint_no_ignored_tests` guard test scans `src/` + `tests/` and fails if the attribute reappears (ignored tests rot silently — three e2e tests drifted out of sync with the UI that way). A test that can't always run gets a **Cargo feature gate** instead: the iOS-Simulator e2e (`tests/e2e_ios.rs`, WebKit rendering checks) is gated behind `--features ios_e2e` (boots a sim, 10–60s, needs Xcode) and run on the mini, not in the default suite.
- `cargo test <test_name>` — single test (e.g. `cargo test roundtrip`).
- `npm run format` — Prettier + Tailwind plugin formatting (templates / styles / config).
- `./build/macos/build.sh [--profile beta|prod]` — builds an ad-hoc-signed app bundle (no Apple Developer ID, no notarization, no `.pkg`). `--profile` (default `prod`) picks the bundle name + identifier so beta and prod coexist in `/Applications` and LaunchServices: `prod` → `Hotchkiss-IO.app` / `io.hotchkiss.web`, `beta` → `Hotchkiss-IO-Beta.app` / `io.hotchkiss.web.beta`. Honors `CARGO_TARGET_DIR`; writes `$CARGO_TARGET_DIR/<name>.app` and prints `BUILT_APP=<absolute-path>` + `PROFILE=<profile>` on stdout. Resolves `VERSION` from `$VERSION` → `$GITHUB_REF_NAME` → `git describe`, falling back to `0.0.0-dev` — on the mini the post-receive extracts a `.git`-less tree, so CFBundleVersion lands on `0.0.0-dev` (cosmetic; the runtime/log/tray version comes from `CARGO_PKG_VERSION`, so bump `Cargo.toml` to match each tag — single-source fix is backlogged). The post-receive hook calls this per profile.

Deployment is push-to-deploy with an **inverted flow** (Phase 12): one bare repo at `~/repos/hotchkiss-io.git` on the Mac mini, whose `post-receive` hook (`build/macos/post-receive`) **dispatches by pushed ref** — `git push origin main` → **beta** (bleeding edge), and `git push origin vX.Y.Z` (a `v*` tag) → **prod** (deliberate promotion). For each deploying ref the hook archives the tree into a per-profile work dir, runs `build/macos/build.sh --profile <profile>`, atomically swaps the new `.app` into `/Applications`, and **gracefully restarts** the matching LaunchAgent — `launchctl kill TERM` (NOT `kickstart -k`/SIGKILL, which made macOS file a "quit unexpectedly" crash report on the mini every deploy, Phase BY); `KeepAlive=true` relaunches it onto the swapped bundle (after up to `ThrottleInterval`, kept as a crash-loop backstop), self-bootstrapping if it isn't loaded. A failed build leaves the running app untouched, and each ref deploys in an isolated subshell so one ref's failure can't skip a sibling. **Releasing to prod:** bump `Cargo.toml`'s `version`, commit, `git push origin main` (lands on beta), then `git tag vX.Y.Z && git push origin vX.Y.Z`. Two LaunchAgents run side-by-side — prod `io.hotchkiss.web` on `:80`/`:443`, beta `io.hotchkiss.web.beta` on `:8080`/`:8443`. `origin` is the mini (the deploy remote); the public `github` mirror (`github.com/chotchki/hotchkiss-io`) was **re-added 2026-06-29** once the docs were de-personalized + the history rewritten to purge personal career context (Phase CM). Publishing is manual — `git push github main` + `git push github $(git tag -l 'v*')`; the mini stays the deploy path and nothing auto-pushes to GitHub. (Note: the mini's pre-CM `v*` tags still point at the OLD pre-rewrite lineage — they were deliberately NOT force-updated, since pushing 35 moved tags to the mini would trigger 35 prod deploys; GitHub carries the rewritten tags. New releases tag from the rewritten `main`, so both stay clean going forward.) Apple notarization / Developer ID / `.pkg` distribution was retired (2026-05): ad-hoc signing is sufficient. One-time mini setup is in `build/macos/SETUP.md`.

**Beta instance** (`beta.hotchkiss.io:8443`) is public (grey-cloud A record that beta's own `DnsProviderService` tracks to the public IP, reached via a router `:8443` forward) and serves its **own** LE-prod cert — beta deploys as a release build, so the cert is natively trusted (the iPhone installs the PWA with no profile). Every `main` push **snapshots prod's `database.sqlite` into beta** (`snapshot_prod_db_into_beta` in the hook, via `sqlite3 .backup` of the live prod DB): it scrubs `crypto_keys` EXCEPT id 2 (beta regenerates its session signing key id 1 but KEEPS the media-URL HMAC key, so its `url_key`s match prod's), `tower_sessions`, `request_log` (visitor privacy), and `instant_acme_domains` (prod's LE account key); **rsyncs prod's media store into beta's own media dir** (content-addressed, `--ignore-existing` so incremental — beta's config sets `media_paths`/`backup_path` under `io.hotchkiss.web.beta/`, since `app_support` is hardcoded to `io.hotchkiss.web`; the rsync iterates prod's `media_paths`); **preserves** beta's own cert + ACME account (captured/restored, so beta never re-orders against the 5/week LE duplicate-cert limit); and **carries prod's `users`/passkeys** so chris's prod passkey authenticates on beta — beta's `webauthn_rp_id` is the registrable parent `hotchkiss.io` (set in beta's config), and the WebAuthn `rp_origin` includes the non-default `:8443` port. Beta data is therefore ephemeral by design: any beta-only edit is blown away on the next `main` push.

## Build-time machinery (build.rs)

`build.rs` does four things that surprise people:

1. **Creates an ephemeral SQLite schema DB at `$OUT_DIR/schema.db`** and runs `src/db/migrations/` against it, then writes `DATABASE_URL` into `.env` and exports it to rustc. This is what lets `sqlx::query!` / `query_as!` macros type-check at compile time. **If you add a migration or change a query, you may need `cargo clean -p hotchkiss-io` (or delete the OUT_DIR `schema.db`) for sqlx macros to re-validate.**
2. **Downloads a pinned Tailwind CLI** (`TAILWIND_VERSION` in `build.rs`, currently `v4.3.0`, arm64 macOS standalone binary) into `OUT_DIR` (cached under a version-keyed filename, so bumping the pin re-downloads) and compiles `styles/tailwind.css` → `assets/styles/main.css`. The compiled CSS is gitignored. (DaisyUI used to be downloaded here too but was never wired into `tailwind.css` — removed 2026-05; the site is styled with hand-rolled Tailwind utilities.)
3. **Generates the inline-SVG icon set** (Phase CN): reads the vendored Font Awesome Free 6.7.2 solid SVGs in `build/svg-icons/solid/` (kept OUTSIDE `assets/`, so `rust-embed` doesn't ship them) and emits `templates/partials/icons.html` — one parameterless askama macro per icon, called as `{% call icons::pen() %}` → inline `<svg class="icon" fill="currentColor">` (`.icon` is `height:1em;width:auto;fill:currentColor` in `tailwind.css`, so it inherits text size/color; wrap a call in a sized `<span>` for a bigger/recolored icon). **FontAwesome is GONE** — the 155 KiB `fa-solid-900.woff2` + ~18 KiB CSS were the single biggest LCP critical-path cost (PSI render-blocking ~1.9s). **To add a UI icon: drop its SVG in `build/svg-icons/solid/<name>.svg` and call `icons::<name_with_underscores>()`** — a typo'd macro is a compile error (no un-vendored icon sneaks in, so there's no template scan). WRITE-IF-CHANGED so an unchanged regen doesn't bump the mtime + cascade a recompile (the partial lands under the `templates` rerun-trigger); the generated `icons.html` is gitignored. The STL-viewer loading spinner is a self-contained `.stl-spinner` CSS ring (no icon, no dep on Tailwind scanning JS).
4. Re-runs only when `assets/scripts`, `templates`, `migrations`, or `build/svg-icons` change.

`assets/` and `templates/` are bundled into the binary at compile time via `rust-embed` (`web/static_content.rs`) and `askama` respectively — there are no loose static files at runtime besides the SQLite database.

## Runtime architecture

`src/main.rs` → `lib::real_main` → `tray-wrapper` (macOS menu-bar anchor) → `create_server` closure → `ServiceCoordinator`.

The coordinator (`src/coordinator/service_coordinator.rs`) wires four long-lived tokio tasks connected by `broadcast` channels:

```
IpProviderService ──IPs──▶ DnsProviderService (Cloudflare API)
                                 │
AcmeProviderService ──RustlsConfig──▶ EndpointsProviderService (HTTP/HTTPS axum servers)
```

- **IpProviderService** polls `https://1.1.1.1/cdn-cgi/trace` hourly (parsing the `ip=` line) and broadcasts the public IP. Connecting to the IPv4 literal forces a v4 path. In `debug_assertions` builds it forces `127.0.0.1` so dev never touches the public DNS.
- **DnsProviderService** updates Cloudflare A records when the IP differs from DNS.
- **AcmeProviderService** loads or orders a Let's Encrypt cert (DNS-01 via Cloudflare), persists it in SQLite (`certificates` table), broadcasts a `RustlsConfig`, and refreshes every 6h. Serves **HTTP/2** (ALPN `h2`, h1.1 fallback; no HTTP/3) — the ALPN policy + the axum-server-needs-ALPN-or-it-silently-stays-h1 gotcha live in `build_server_config` and its unit test.
- **EndpointsProviderService** waits for a `RustlsConfig` then starts both an HTTP→HTTPS redirect server on `:80` and the real axum app on `:443` (served with `into_make_service_with_connect_info::<SocketAddr>()` so handlers/middleware can see the client IP). It also owns the `tower-sessions-sqlx-store` session GC task, a daily `request_log` prune task (90-day retention), and a **daily DB backup task** (`coordinator/backup.rs`): each tick `VACUUM INTO`s a dated snapshot (`database-YYYY-MM-DD.sqlite`) under `Settings::backup_path`, then prunes to a rolling 7 most-recent. The backup loop never returns and matches+logs every fallible step (`tracing::error!`) so a backup failure can't bubble out and trip the coordinator's `try_join!` (which would take the whole app down) — a bad day self-heals on the next tick. Backups exist only on disk; Backblaze covers off-site.

If any task returns, `tokio::try_join!` fails the whole coordinator — there is no auto-restart.

## Web layer (`src/web/`)

- **Routing** is composed in `web/router.rs`: `/` redirects to the first content page, `/login`, `/pages`, `/projects`, `/blog`, `/admin`, `/media` are nested routers (`/attachments` was retired in BZ.8), and `static_content()` is merged in. A custom **404** (`web/features/not_found.rs`) — the "blame the cat" page (`templates/404.html`: three cat suspects, tap-to-accuse quip overlay via a pure-Tailwind native-`<details>` `group-open:` reveal, NO JS and NO custom CSS, same no-JS approach as the nav hamburger; cat photos are web-optimized AVIF under `assets/images/404/`) renders from ONE `render_not_found` helper wired to BOTH the global router `fallback` (unmatched routes; attached **before** `with_state` so it can build the nav) AND the `get_page_path` `/pages/<slug>` miss branch (so a dead link like `/pages/Resume` lands there too — the mutating `/pages` handlers keep their plain `"No such page"`). It returns a real `404` by wrapping `HtmlTemplate` (which always renders `200`) in `(StatusCode::NOT_FOUND, _)`; a nav-build DB failure degrades to a plain 404, never a 500. `LiveReloadLayer` is added **only** in debug builds. The outer `ServiceBuilder` stack adds (outermost first) a request-logging middleware (`web/middleware/request_log.rs` → `request_log` table, fire-and-forget insert), the `TraceLayer`, the session layer, and compression. The admin `/admin/analytics` dashboard (`web/features/admin/analytics.rs`) aggregates `request_log` over a date-range window (`?since=` days, `?metric=total|unique`) — a server-rendered inline-SVG views/day chart, a total↔unique-visitors toggle, top pages with a **Content↔All toggle** (`count_by_content_path` takes a `max_status` ceiling: Content = `status < 400`; All raises it to surface 4xx/5xx scanner probes — static prefixes excluded either way), and **top external referrers** (`count_by_referer`: non-null, self-host filtered, directional only). On-the-fly `GROUP BY` over the 90-day window, no rollup table.
- **SEO + unified feed (Phase CB).** The Atom feed is now ONE feed carrying blog posts AND project pages, newest-first, retitled to chris's name — served canonically at `/feed.xml` (the `<link rel="alternate">` points here) with `/blog/feed.xml` kept as a back-compat alias to the SAME handler (`web/features/feed.rs`; entries carry a `<category term="blog|projects">` and link to `/blog/<slug>` vs `/pages/projects/<slug>` — project DETAIL pages live under the `/pages` content tree; the `/projects` route is the index ONLY (Phase CD fix — they were wrongly linked at a dead `/projects/<slug>`)). `/sitemap.xml` + `/robots.txt` are now DYNAMIC (`web/features/seo.rs`, registered top-level in `router.rs`; the static `robots.txt` asset + its `static_content` route were dropped): the sitemap lists home + top-level `/pages/<slug>` + the blog/projects indexes and their children (blog → `/blog/<slug>`, projects → `/pages/projects/<slug>`) + `/resume` (absolute `<loc>`s from the request host so they're same-origin as the sitemap, `<lastmod>` = `page_modified_date`; special redirect rows are mapped to their real routes, never leaked as `/pages/<slug>`). **The request host is resolved via `web/util/host::request_host(headers, uri)` — the `Host` header OR, falling back, the HTTP/2 `:authority` (which is where the host lives on an h2 request; there is NO `Host` header).** GOTCHA fixed in Phase CD: enabling h2 (v0.0.69) silently broke this — `headers.get(HOST)` returned `None` over h2, so the live sitemap/robots/feed emitted `https://localhost/…` to every real (h2) crawler, AND beta stopped de-indexing (host→`localhost`→"canonical"). Don't reach for `headers.get(HOST)` directly; use `request_host`. `robots.txt` is **host-aware**: on the canonical host it `Allow: /` + hides `/admin/` + `/login/` + emits the `Sitemap:` directive; on ANY non-canonical host (beta) it `Disallow: /` so the ephemeral beta copy never competes with prod for the same content (duplicate-content is exactly what suppresses crawling). Per-page `<meta name="description">` + `<link rel="canonical">` + OpenGraph/Twitter tags come from a `Meta` struct (`seo.rs`) rendered by `templates/partials/seo_meta.html` (an `{% include %}`'d `{% block meta %}` override on get_page / blog-index / projects-list; base.html carries a site default for every other page) — description from the markdown excerpt (fallback: site tagline), `og:image` from the page cover (fallback: the site photo). **Canonical/OG URLs are built from the canonical `site_host` (`hotchkiss.io`), NOT the served host**, so a beta page's canonical points at prod (dedupe). The prod "analytics doesn't work on mobile" report was a base-layout bug, not analytics-specific: a wide unwrapped table forced the whole document past 390px (the logged-in nav contributed too), and Safari mangled the layout. Fixed by capping the main content column at the viewport (`base.html`: `p-2 w-full min-w-0` — a definite width + `min-w-0` so the flex child can't shrink-wrap to a wide descendant) **so the per-table `overflow-x-auto` wrappers actually scroll instead of growing the page**. **Manual step:** after publishing content that should re-index fast, resubmit `/sitemap.xml` in Google Search Console — nothing auto-pings.
- **Templating**: askama templates in `templates/`, served via the `HtmlTemplate<T>` wrapper. Templates reference `BUILD_TIME_CACHE_BUST` (a compile-time UTC epoch) for asset cache-busting query strings. **Static-asset caching (Phase CN):** `static_content` serves a versioned (`?cb=`) request `Cache-Control: public, max-age=31536000, immutable` (the token IS the build epoch, so a stale cache can never serve wrong bytes) and an un-versioned bare fetch (a raw `/favicon.ico`) the 1-day TTL; the policy is the pure, unit-tested `cache_control(debug, versioned)` (debug = `no-store` for live-reload). **htmx is `defer`'d** off the LCP critical path — every script that calls `htmx.*` at parse time (`htmx-webauthn`, `htmx-sortable`; the stl-view `type=module` already defers) MUST also `defer` so it runs after htmx (defer preserves document order). The self-hosted `@font-face` (Oswald, Quattrocento) carry `font-display: swap` (paint immediately, no FOIT).
- **HTMX**: the frontend uses HTMX (`assets/vendor/htmx/`). Mutating handlers return `htmx_refresh()` or `htmx_redirect(...)` from `htmx_responses.rs` rather than rendering a response body.
- **Sessions**: `tower-sessions` backed by SQLite, signed with a key from the `crypto_keys` table (auto-generated on first boot via `CryptoKey::get_or_create`). Expiry: 1 day inactivity, secure-only.
- **Auth**: WebAuthn / passkeys via `webauthn-rs` (discoverable credentials, conditional UI). State machine in `AuthenticationState`: `Anonymous → AuthOptions → Authenticated`, or `Anonymous → RegistrationStarted → Authenticated`. Registration first creates the `UserDao` with `Role::Anonymous`; the SQL `INSERT` then conditionally upgrades to `Admin` (first user) or `Registered`.
- **API-key auth (Phase CA)**: an `Authorization: Bearer hio_…` key authenticates as its user, **delegating that user's role** (full delegation — a key minted by an Admin has Admin). The `api_key_auth` middleware (`web/middleware/api_key_auth.rs`, `from_fn_with_state`, layered OUTER to the authz layer) resolves the key and injects an Authenticated `SessionData` into the request; the `SessionData` extractor reads that injection FIRST, so the existing fail-closed layer + every handler see the key's user with ZERO handler changes (no prebuilt DB-backed-key middleware fits — `tower_http::ValidateRequestHeaderLayer` is single-static-token + sync only). Keys are user-managed at `/admin/api-keys` (generate → plaintext shown ONCE, list with last-used, revoke instantly — scoped so you only see/revoke your own). Stored as **HMAC-SHA256(server pepper = `crypto_keys` id 3, key)** hex — the key itself is never stored, so a DB leak can neither recover nor verify it; live until revoked (`api_keys` table, migration `0016`; `hio_<base64url>` from 32 random bytes). **Env-specific:** the beta snapshot scrubs `crypto_keys` id 3 (only id 2 is preserved), so prod keys don't authenticate on beta and the carried `api_keys` rows are inert there — generate a beta-specific key if you need one.
- **User management (Phase CC)**: `/admin/users` (`web/features/admin/users.rs`, admin-gated by the `/admin` nest's `require_admin`) lists every user (role + passkey count = `json_array_length(keys)` + live API-key count) and promotes/demotes between `Registered`↔`Admin` (`POST /admin/users/{id}/role`) or deletes (`DELETE /admin/users/{id}`, which wipes the user's `api_keys` first — FK — then the row, in a tx; passkeys live in the `keys` column and go with it). **The last admin is protected** from demote/delete (no lockout) — enforced in the handlers (`409`) AND hidden in the UI; self-actions are otherwise allowed. There's still **no create-user** path: users self-register via passkey (first registrant → Admin). **Live enforcement** — a cookie session stores a SNAPSHOT of the user (`Authenticated(UserDao)`, role included), so a role change / delete wouldn't bite a live session until logout/expiry. The `refresh_session_role` middleware (`web/middleware/refresh_session_role.rs`, `from_fn_with_state`, layered INNER to `api_key_auth` / OUTER to the authz layer) re-loads the cookie session's user by id each request and injects a refreshed `SessionData` (current role) — or Anonymous if the user was deleted — so demote/delete take effect IMMEDIATELY (a Bearer key already resolves fresh, so the middleware no-ops for it). One extra cheap SQLite lookup per authenticated request.
- **Cookie hardening (Phase CC)**: the session cookie — the ONLY cookie the app sets (tower-sessions) — is explicitly `HttpOnly` + `SameSite=Lax` + `Secure` (release only; `with_secure(!debug)` keeps it off for the plain-HTTP test harness / dev). `HttpOnly` is tower-sessions' default, set explicitly so it can't silently regress; an integration test asserts `Set-Cookie` carries `HttpOnly` + `SameSite`.
- **Authorization is one fail-closed layer** (`web/middleware/require_admin_for_mutations.rs`, Phase E): **GET/HEAD/OPTIONS are public site-wide; every other method requires an authenticated Admin BY DEFAULT.** It's a default-DENY allowlist of safe methods (not a deny-list of known-mutating verbs), so a newly-added route or an exotic verb (e.g. the `/pages/preview` PATCH) is gated automatically — the secure default protects against the per-handler-omission bug it replaced (v0.0.49: an anonymous caller could mutate `/pages`). The only exceptions are an EXACT `(path, method)` allowlist of the anonymous WebAuthn ceremony POSTs (`/login/finish_authentication`, `/login/finish_register`) + the debug `/test/login` seam. Wired in `router.rs` **inner to the session layer** (so `SessionData` is loaded; it defaults to `Anonymous` → a clean 403, not a panic). The **`/admin` nest keeps its own `require_admin` layer** (`web/middleware/require_admin.rs`) — it gates a *non-public* GET (`/admin/analytics`) that the GET-public default would otherwise expose; the two are complementary. **No per-handler authz checks remain** — the layer is the single source of truth for mutations. (`list_page_attachments` GET was made public in Phase F: the attachment *content* is already unauthenticated, so the listing leaks nothing.)
- **Errors**: handlers return `Result<Response, AppError>`. `AppError` wraps `anyhow::Error`, logs with a UUID trace id, and returns a 500 with that id to the client.
- **`AppError` swallows status info** — if a handler needs to return e.g. a 404 or 403, return `Ok((StatusCode::X, "msg").into_response())` instead of bubbling via `?`. This is a deliberate pattern, see existing handlers for examples.

## Content model

Pages are a self-referential tree in `content_pages` (parent_page_id → page_id). Lookup by URL path goes through `ContentPageDao::find_by_path(&[&str])` which walks segments. **Special pages** (`special_page = true`, seeded by migrations `0007`, `0010` and `0012`) are routing redirects — their `page_markdown` is treated as a redirect target URL, not Markdown. `login`, `projects`, `blog` and `resume` are the current special pages; they cannot be deleted. The `/projects` and `/blog` handlers each query their special page by name and list its children — a misnamed/missing special page surfaces as a 500 with "Server misconfiguration". `/blog` (`web/features/blog.rs`) lists posts as cards newest-first by `page_creation_date`; the Atom feed moved to the unified `/feed.xml` (Phase CB) with `/blog/feed.xml` kept as an alias. **Pagination + search (Phase CH):** both `/blog` (newest-first) and `/projects` (manual `page_order`) paginate (`PAGE_SIZE=10`, `?page=N` 1-indexed, prev/next at the bottom, omitted at the ends) and support `?q=` text search over title + markdown + slug — they share `web/features/listing.rs` (`Pagination` + the `paginate(pool, parent, query, ListOrder, base_path)` helper) backed by `ContentPageDao::count_children` / `find_children_{newest,ordered}_paged` (a `LIKE '%q%'` filter; **FTS5 is the noted upgrade if the corpus grows**), and the `partials/listing_{search,pager}.html` partials (no-JS GET form + links; the two compose as `?q=…&page=N`). Each page keeps its own card type + ordering; only the windowing/search machinery is shared. **The Atom feed stays full** (newest 50, unpaginated). **Editable post date (Phase CE):** `page_creation_date` is editable — the editor's "Posted" `datetime-local` field + the `page_creation_date` PUT form field (parsed by `parse_local_datetime`, empty/unparseable → keep existing) let a Wayback-recovered old post be **backdated** to its real chronological slot. `ContentPageDao::update` now writes `page_creation_date` (it didn't before); `page_modified_date` still auto-stamps `now` on every save. The post page (`get_page.html`) shows the date under the title for blog posts (`GetPageTemplate.posted_date` = `Some` only on `/blog/<slug>`; `None` on `/pages` + `/resume`). `/blog/<slug>` renders a post via the same `GetPageTemplate` as `/pages/...` — the editor form posts back to `/pages/<page_path>` absolutely. **Next/previous nav (Phase BX):** a blog post ends with Previous (older) / Next (newer) cards to the chronologically adjacent posts — `show_post` locates the post in `find_by_parent_newest_first` (same order as the index) and takes its neighbors, carried as `Option<PostNavCard>` `prev_post`/`next_post` on `GetPageTemplate`; `/pages` passes `None` so the nav is blog-only by construction, and a side is omitted at the ends. **Résumé (Phase 16):** `resume` is a special page (migration `0012`) whose newest child holds the résumé; the dedicated `/resume` route (`web/features/resume.rs`) renders that child via the same `GetPageTemplate` (with `pdf_url` set → a Download-PDF button), and `/resume.pdf` renders the same markdown to HTML, wraps it in a print stylesheet (`resume-print.css`) and shells out to **`weasyprint`** (resolved `$WEASYPRINT_BIN`→brew→PATH, cached by content hash) — single source, the PDF derives, no drift. The PDF HTML carries a `<base href="https://{site_host}/">` (`site_host` = the WebAuthn rp-id — `hotchkiss.io` on prod AND beta, the canonical public host, NOT the served domain) so the root-relative links `rewrite_site_links` stores resolve to **absolute, clickable** URLs in the downloaded PDF — without it weasyprint emits dead `file:///path` link annotations (empirically verified); a résumé generated on beta therefore still points recruiters at the real `hotchkiss.io`. `weasyprint` must be installed (dev + the mini), like `d2`; a missing/broken binary logs + returns a 500 (a PDF download can't show an inline error block). chris authors the résumé at `/resume?edit`. **Title vs slug (Phase F):** `content_pages.page_title` (migration `0011`, nullable) is the human title; `page_name` stays the URL slug. `ContentPageDao::display_title()` = `page_title ?? first markdown H1 ?? page_name`, used for blog cards, breadcrumbs, the page `<h1>`, the `<title>`, and the Atom feed; the leading `# H1` is stripped from the rendered body (`web/markdown/title::strip_leading_h1`) so the title isn't shown twice. Page creation takes a *title* and auto-slugs `page_name` (`web/util/slug::slugify`), then htmx-redirects to the new page in edit mode. **Admin authoring (Phase F):** an admin defaults to the clean reader view; `?edit` reveals the editor (an "Edit this page" button toggles it) so admin can browse the real site without logging out. Login state + Logout + the new-page form live in a dedicated **admin bar** below the nav (not crammed into the content tabs), and the LOGIN tab hides when authenticated.

Markdown is rendered through `web/markdown/transformer.rs`, which rewrites `.stl` image links to `<object class="stl-view" ...>` tags so the frontend STL viewer can pick them up, rewrites every **other** image link to a height-capped (480px) click-to-zoom `<img>` (`data-zoomable` — reusing the diagram lightbox, so a tall screenshot doesn't dominate the page), turns fenced **```d2** code blocks into inline diagrams, and short-circuits an `![](/media/<ref>)` link to the media-embed dispatch (Media, below). The diagram pipeline (`web/markdown/diagram.rs`) is **source-in-HTML + HTMX swap** (Phase A): at page-render the fence becomes a one-line placeholder that shows the **d2 source** in a `<pre>` (so a crawler / LLM / no-JS reader sees the real source — diffable, LLM-parsable) and carries `hx-get="/diagram/<hash>" hx-trigger="load" hx-swap="outerHTML"`; the source is registered keyed by a SHA-256 content hash (content-addressed — distinct diagrams can't collide, identical ones dedupe). On load HTMX GETs the public `/diagram/{hash}` route (`web/features/diagram.rs`), which shells out to the **`d2` binary** (`brew install d2`; resolved via `$D2_BIN` → `/opt/homebrew/bin` → `/usr/local/bin` → PATH, so it works under the mini's minimal LaunchAgent PATH), embeds the SVG as a base64 `data:` URI `<img>` (isolated → no id/font collisions across diagrams), caches it, and returns it for the swap. The endpoint renders **only sources the server already emitted** (looked up by hash) — not an open d2 compiler. A missing d2 or a bad source returns a visible error block at HTTP 200 (so HTMX still swaps), never a 500. **`d2` must be installed** on dev + the mini + CI; without it, diagrams degrade to the error block. In-flow diagrams are capped at a max-height (the natural SVG size is injected so the `<img>` scales proportionally), and clicking one opens a zero-dependency pan/zoom lightbox (`assets/scripts/diagram-zoom.js`, loaded in `get_page.html`, bound via event delegation on any `img[data-zoomable]` so it works on both HTMX-swapped-in diagrams AND content images — pattern borrowed from recon-gen's `qs-lightbox`; CSS lives in `styles/tailwind.css`). Adding new AST rewrites means extending the `match` in the BFS walk in `transformer.rs`. **On save** (not render), `web/markdown/links.rs::rewrite_site_links` rewrites absolute links + image `src`s that point at the site's own **registrable** host (`webauthn_rp_id` — `hotchkiss.io` on both prod and beta, surfaced via `AppState.site_host`; sourced from the rp-id and NOT the served `domain` precisely so beta's `beta.hotchkiss.io` still relativizes the canonical `hotchkiss.io` links its DB snapshot carries — plus the `www.` variant, any scheme/port) to root-relative, preserving query + fragment, so authored content is portable across prod/beta/any host. It's wired into `put_page_path` and replaces only the matched URL strings (longest-first, so a bare-domain match can't corrupt a longer path URL), leaving the rest of the author's markdown formatting byte-for-byte intact. **Typeset math + code highlighting** (Phase BV): the transformer enables markdown-rs's `math_text`/`math_flow` constructs with **single-dollar OFF** (so `$$…$$` is math but a prose `$200` price stays literal), and emits each math node as a source-carrying `<span class="math math-inline">` / `<div class="math math-display">` (the TeX is in the served HTML — no-JS/crawler/LLM reads it); **KaTeX** (`assets/vendor/katex/`, init `assets/scripts/katex-render.js`) typesets the `.math` elements client-side on load + after HTMX swaps. Fenced code keeps its `language-*` class and **highlight.js** (`assets/vendor/highlightjs/`, init `assets/scripts/code-highlight.js`) highlights `pre:not(.d2-source) code` (the d2 source is excluded). Both load in `get_page.html` and degrade to readable source without JS. **(Phase BW)** the BFS walk descends into ALL containers (lists, headings, blockquotes, emphasis, …), so math/images/diagrams nested anywhere convert — not just at top level (a Root/Paragraph-only walk was a BV regression that left list-nested math as raw `$…$`). **GFM tables** are enabled (`gfm_table`); since `mdast_util_to_markdown` (0.0.1) can't re-serialize a `Table` node, each table is rendered from its original markdown source slice in the walk and emitted as HTML (bypassing the AST round-trip that would otherwise drop it; markdown-rs `Point.offset` is a BYTE offset, so the slice is char-safe). **Panic-hardened (Phase CG):** the alpha `markdown-rs` / `mdast_util_to_markdown` can PANIC on edge-case content — a Wayback-recovered 2012 post (smart quotes + escaped angle brackets) did, and since an uncaught handler panic RESETS the connection (a `000`, no 500) AND the feed transforms EVERY post, that one post took down its page *and* the whole feed. Two layers fix it: `transform()` wraps the work in `std::panic::catch_unwind` and degrades to the escaped source on panic (so a page/feed always render something safe), and a router-level `CatchPanicLayer` (`router::handle_panic`, outermost) turns ANY handler panic into a styled 500 instead of a dropped connection.

**Media (Phase BZ).** ALL binary media — images, video, STLs — live in a content-addressed **disk store** (`MediaStore`, `src/media/`), NOT SQLite BLOBs: files under the configured `Settings.media_paths` roots (default a single `app_support/media`) sharded `ab/cd/<sha256>`, atomic temp+rename, dedup by content. Schema: `media`(`media_ref` UNIQUE, kind, title, dims, duration) + `media_variant`(sha256, `url_key` UNIQUE, mime, codecs, bytes) — one item, N variants (a video's AV1 + HEVC encodes + an auto-poster image). **Two URLs, both unguessable:** the author ref `/media/<media_ref>` is an opaque **UUIDv7** token (NOT a slug — closes the slug-guess oracle); the byte route `/media/file/<url_key>` where `url_key` = **HMAC-SHA256**(server key — `crypto_keys` id 2, NOT the id-1 session key — , sha), served by `tower_http::ServeFile` (HTTP range / 206, immutable cache) and looked up by the *stored* `url_key` (so a key mismatch across prod/beta still serves — the file resolves by sha). **(Phase CL security)** the byte route always sends `X-Content-Type-Options: nosniff` and forces `Content-Disposition: attachment` on executable mimes (`html`/`svg`/`xml`/`js`) — a generic `MediaKind::File` carries a filename-guessed mime and the route is public + same-origin, so an admin-uploaded `.svg`/`.html` must NOT render as active script on the canonical origin (stored XSS); probe-verified image/video/stl render inline. The path resolve also runs in `spawn_blocking` so a stat to an asleep external drive can't pin a tokio worker. `title` is the human name (library display + search). **Ingest** (`/admin/media`, admin-gated): `ffprobe` derives codec/mime/dims/duration (`.stl` by ext; image-vs-video by `format.duration`); a video gets an **auto-poster** (ffmpeg frame → AVIF). `ffprobe`/`ffmpeg` must be installed (dev + mini), like `d2`/`weasyprint`. **Large-file streaming + generic files (Phase CI):** uploads STREAM to disk — `upload_media`/`add_encode` pull the multipart field chunk-by-chunk (`field.chunk()`) through `MediaStore::stage` → `StagedBlob` (writes to a temp in `<media_path>/.staging`, hashes incrementally with `sha2` — openssl's `Hasher` isn't `Send` so it can't cross the `.await`, but the digest is identical so dedup stays consistent with the `store(&[u8])` path — then atomic-renames into `ab/cd/<sha>`, or dedups + drops; an aborted upload self-cleans on `Drop`). So a multi-GB file is **disk-bound, not RAM-bound** — the old `field.bytes()→Vec<u8>` buffered the WHOLE upload and OOM'd around free-RAM size. (`DefaultBodyLimit::disable()` is already on both upload routes; there's no request-timeout layer.) A file `ffprobe` can't type — zip / app / binary, anything non-image/video/STL — ingests as **`MediaKind::File`** (mime guessed from the extension, octet-stream fallback) and renders as a download `<a>`, rather than the upload being rejected; a **missing/un-spawnable ffprobe still errors loudly** (that's a deploy misconfig, not a non-media file). The library's **"Copy link"** button copies the absolute `/media/file/<url_key>` — a direct, unguessable share/download URL (downloads already stream via `ServeFile` + range). The `.staging` dir is `--exclude`d from the prod→beta media rsync. **Multi-drive (Phase CJ):** `media_paths` is an ORDERED list of roots (`media_min_free_bytes` headroom, default 10 GiB). Uploads fill IN ORDER — `MediaStore::stage`/`pick_write_root` picks the first root with headroom UP FRONT (so the commit rename stays intra-volume; a cross-volume rename is `EXDEV`) and falls to the next when a drive drops below the margin (the streamed size isn't known up front). **(Phase CL hardening)** a root is a write target ONLY if **present** — its dir OR its parent exists, i.e. the drive is mounted; a cleanly-unmounted external root (macOS removes its `/Volumes/<name>` entry on eject) is skipped, so `pick_write_root` never `create_dir_all`s a phantom mountpoint chain onto the boot disk (the M1 bug) and uploads fall THROUGH it — a per-root stat error falls through too (L1: it no longer aborts the whole upload). `roots_status` shares the same `probe_root`, so the storage panel and the writer always agree. **Configure a SUBDIR under each volume (NOT the mount root)** so the present-check is meaningful; the free-space margin is the backstop against the rare unclean-unmount stray-`/Volumes/<name>`-dir case. Each variant records its `storage_root` (migration `0017`, nullable HINT); `resolve_path(sha, hint)` tries that root first (O(1)) then first-found scans all roots — self-healing if a file moved, and a NULL hint (legacy rows) just always scans. Commit dedups across ALL roots (identical bytes don't double up on a second drive). An unmounted root → that variant 404s (keeping roots mounted is the operator's job); Backblaze covers backup at the filesystem level across every drive, so the app never copies media for backup. The prod→beta rsync reads prod's `media_paths` from its config + merges each root into beta's one media dir. **Config migration (no back-compat): rename beta's `media_path` → `media_paths` (a one-element array) before deploying CJ to beta** — else it silently falls to the default media dir (= prod's). Prod uses the default unless you ADD drives to its `media_paths`. **Upload progress (Phase CK):** every upload path POSTs via `XMLHttpRequest` (not `fetch`, which exposes no upload progress) through the shared `assets/scripts/upload-progress.js` (`UploadProgress.xhrUpload` + `fmtBytes`) → a native `<progress>` bar + "Uploading X% — loaded/total" that flips to "Processing…" once the body's fully sent (the bar tracks bytes SENT, ≈ all the wall-clock for a big file); server unchanged. **Authoring is uniform** `![alt](/media/<ref>)` for EVERY kind: the transformer rewrites a `/media/` link to `<span hx-get="/media/embed/<ref>" hx-trigger="load" hx-swap="outerHTML">`, and `render_media_embed` dispatches by kind → `<img data-zoomable>` / `<video>` multi-`<source>` / `<object class="stl-view">`. **Responsive images (Phase CN):** an image upload ALSO generates width-stepped AVIF downscales (`src/media/resize.rs`, widths 480/960, skipping any ≥ source) — each a content-addressed `image/avif` `media_variant` carrying its pixel `width` (migration `0018` added `media_variant.width`/`height`; `media.width/height` is the original ITEM's dims, this is per-ENCODING). `render_embed_html` then emits `<img src srcset="…480w,…960w,…<orig>w" sizes="(max-width:768px) 100vw,768px">` (src = the largest, for a no-srcset client + the zoom view); `cover_url_for` serves the SMALLEST width-stepped variant for a ~300px card thumbnail. A legacy image with no widths degrades to a single `<img src>`. A **background, idempotent startup backfill** (`coordinator/backfill_responsive_images.rs`, spawned DETACHED — never in the `try_join!`, so it can't delay boot or take the app down; backup-first, per-item non-fatal) generates the downscales + stamps the original's width for images uploaded before CN. The resize reuses poster.rs's in-process `image`-crate AVIF path (no new ffmpeg-build dep); rav1e is slow, so the ladder is capped at 960 + uploads run it best-effort under `spawn_blocking`. **Video sources are ordered HEVC-before-AV1** (`codec_rank`) — Safari's `<video>` AV1 path is jerky even on AV1-HW Macs; HEVC hardware-decodes smooth (AV1 stays as the non-Apple fallback). The **library** (`web/features/admin/media.rs`, `templates/admin/media.html`, `media-upload.js`): drag-drop grouped upload (drop a video's encodes together → one item), per-card add-encode / rename / per-stream delete / inline play / copy-ref / name search. **Inline editor upload** (`editor-support.js`): drop files on the markdown box (or the 🎞 toolbar button) → async POST `/admin/media/upload` → insert `![](/media/<ref>)` at the cursor with NO page refresh (the old attachment upload `htmx_refresh()`'d and ate unsaved edits). **Covers are media:** `content_pages.page_cover_media_id` (migration `0014`); blog/project cards render via `media::cover_url_for(page_id)` (the SMALLEST width-stepped image variant — a video's poster; Phase CN); the editor cover field takes a media ref (`cover_ref_for` pre-fills, `put_page` resolves → media_id). The `CompressionLayer` excludes `video/`/`audio/`/`model/`/`application/octet-stream` (gzipping a range response corrupts it).

**The old SQLite-BLOB `attachments` are RETIRED** (Phase BZ.8, shipped prod v0.0.63→v0.0.64). A one-shot startup migration (`coordinator/migrate_media.rs`, now removed) copied every BLOB into the store + `media` rows, rewrote page markdown `/attachments/…`→`/media/<ref>` (both URL forms), and re-homed covers — backing the DB up FIRST, deferring if that failed. Then `0015` emptied the table and the `/attachments` route + `AttachmentDao` were deleted. The **prod→beta snapshot** preserves `crypto_keys` id 2 and rsyncs prod's media into beta's OWN media dir (beta's config sets `media_paths`/`backup_path` to `io.hotchkiss.web.beta/`, since `app_support` is hardcoded to `io.hotchkiss.web`). **Leftover:** the now-empty `attachments` table + the dead `page_cover_attachment_id` column intentionally STAY — `content_pages ↔ attachments` is a circular FK (+ a `content_pages` self-FK), so a clean `DROP` needs a table rebuild with `foreign_keys=OFF` that a transactional sqlx migration can't do (and dropping the table while the FK remains makes every `content_pages` query fail to prepare). A future `-- no-transaction` migration can rebuild it away.

## Configuration

`Settings` is loaded from JSON, in this order:

1. First CLI arg, if present (this is what `bacon` and dev use: `data/config.json`).
2. Otherwise `~/Library/Application Support/io.hotchkiss.web/config.json` on macOS, `$HOME/...` elsewhere.

Required fields: `cloudflare_token`, `domain`. Optional path fields — `database_path`, `log_path`, `cache_path`, `backup_path` — default to `~/Library/Application Support/io.hotchkiss.web/data/database.sqlite`, `~/Library/Logs/io.hotchkiss.web`, `~/Library/Caches/io.hotchkiss.web`, and `~/Library/Application Support/io.hotchkiss.web/backups` respectively when omitted (see `Settings::resolve` in `src/settings.rs`). `backup_path` is where the daily DB snapshots land (created on first backup if missing). The on-disk JSON is deserialized into a private `RawSettings`; unknown keys are silently ignored. The `data/` directory in the repo is gitignored and contains the dev config + SQLite database.

The macOS app is **not sandboxed** (the `com.apple.security.app-sandbox` entitlement was dropped, 2026-05) — `NSHomeDirectory()` returns the real home, so the paths above are literal. On the production Mac the app runs as a LaunchAgent (`build/macos/io.hotchkiss.web.plist`) in the user GUI session, which keeps the tray icon alive; macOS Mojave+ lets the non-root process bind ports 80/443 because axum binds `INADDR_ANY`.

## Things to watch out for

- **Vendored OpenSSL**: `openssl` is pinned with `features = ["vendored"]` to avoid system-OpenSSL packaging issues.
- **Ports default to 80/443 but are configurable** (`http_port` / `https_port` in `Settings`, since Phase 11; beta runs 8080/8443) — running locally as non-root on 80/443 will fail to bind, so set high ports in your dev config or accept the bind failure. (On the production Mac 80/443 work unprivileged because macOS Mojave+ allows non-root binds to ports <1024 when binding `INADDR_ANY`.)
- **`assets/styles/main.css` is generated** — never edit it; edit `styles/tailwind.css` and let `build.rs` recompile.
- **Pushing deploys (inverted flow, Phase 12)** — `git push origin main` rebuilds + restarts **beta**; only a `v*` **tag** push rebuilds + restarts **prod** (~10–15s downtime; the restart is a graceful `launchctl kill TERM` + `KeepAlive` relaunch — Phase BY — not a SIGKILL). Prod no longer auto-updates from `main`. A failed build aborts before the swap (the running app keeps serving), but a *successful build of broken runtime behavior* goes live on whichever target the ref maps to. `origin` is the mini (deploy); `github` is the public mirror, re-added 2026-06-29 after the Phase CM docs-scrub + history rewrite — push `main`/`v*` tags there manually to publish (nothing auto-pushes).
- CI layout: `test_and_coverage.yml` runs on `macos-latest` per push (lint + test + grcov coverage), `check_ip.yml` is a small scheduled IP check — both are informational and gate nothing. The `release.yml` / `install.yml` GitHub Actions workflows were deleted (2026-05); deployment is the `git push origin main` → `post-receive` hook flow described under "Common commands". Both Actions went dormant 2026-06 (no GitHub remote); with the mirror re-added 2026-06-29 (Phase CM) they run again on each manual `git push github` — still informational, gating nothing.
