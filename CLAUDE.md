# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Christopher Hotchkiss's personal site / CRM. A single self-contained Rust binary that ships as a macOS `.app` with a tray icon, serves the site over HTTPS, and self-manages its own DNS (Cloudflare) and TLS certificates (ACME via Let's Encrypt). User roles: `Anonymous` (read-only) / `Registered` (read-only, logged in) / `Family` (trusted household tier, promotion-only â€” the audience for the upcoming `min_role`-gated Library; Phase CZ) / `Admin` (edit). The very first user to register is automatically promoted to `Admin` (enforced inline in `UserDao::create`); everyone after lands at `Registered`. The trust ladder is the explicit `Role::rank()` (Anonymous 0 < Registered 1 < Family 2 < Admin 3) â€” **never `derive(Ord)` on `Role`** (variants are declared alphabetically; a derived order ranks Admin below Anonymous â€” a `Role::iter()`-pinned test guards the ladder), and viewer-role derivations go through `AuthenticationState::role()`, not inline matches.

## Build-time machinery (build.rs)

`build.rs` does five things that surprise people:

1. **Creates an ephemeral SQLite schema DB at `$OUT_DIR/schema.db`** and runs `src/db/migrations/` against it, then writes `DATABASE_URL` into `.env` and exports it to rustc. This is what lets `sqlx::query!` / `query_as!` macros type-check at compile time. **If you add a migration or change a query, you may need `cargo clean -p hotchkiss-io` (or delete the OUT_DIR `schema.db`) for sqlx macros to re-validate.**
2. **Downloads a pinned Tailwind CLI** (`TAILWIND_VERSION` in `build.rs`, currently `v4.3.0`, arm64 macOS standalone binary) into `OUT_DIR` (cached under a version-keyed filename, so bumping the pin re-downloads) and compiles `styles/tailwind.css` â†’ `assets/styles/main.css`. The compiled CSS is gitignored. (DaisyUI used to be downloaded here too but was never wired into `tailwind.css` â€” removed 2026-05; the site is styled with hand-rolled Tailwind utilities.) **`@source` gotcha:** Tailwind only scans the dirs listed as `@source` in `tailwind.css` â€” currently `templates`, `src/web/markdown`, `src/web/features`, `assets/scripts` (first-party JS builds player/upload DOM with utility classes; added in DD â€” before that its classes only worked by coinciding with template-used ones) â€” with matching `rerun-if-changed` in `build.rs`. Utility classes emitted from **Rust string builders** (the media embeds in `web/features/media.rs`, the markdown transformer) only get generated because those dirs are scanned â€” **a class used ONLY in a `.rs` string outside a scanned dir silently produces no CSS** (the class name has to appear as literal text somewhere Tailwind reads). Add a new emitter's dir to `@source` + `rerun-if-changed`, or the styles just won't exist at runtime.
3. **Generates the inline-SVG icon set** (Phase CN): reads the vendored Font Awesome Free 6.7.2 solid SVGs in `build/svg-icons/solid/` (kept OUTSIDE `assets/`, so `rust-embed` doesn't ship them) and emits `templates/partials/icons.html` â€” one parameterless askama macro per icon, called as `{% call icons::pen() %}` â†’ inline `<svg class="icon" fill="currentColor">` (`.icon` is `height:1em;width:auto;fill:currentColor` in `tailwind.css`, so it inherits text size/color; wrap a call in a sized `<span>` for a bigger/recolored icon). **FontAwesome is GONE** â€” the 155 KiB `fa-solid-900.woff2` + ~18 KiB CSS were the single biggest LCP critical-path cost (PSI render-blocking ~1.9s). **To add a UI icon: drop its SVG in `build/svg-icons/solid/<name>.svg` and call `icons::<name_with_underscores>()`** â€” a typo'd macro is a compile error (no un-vendored icon sneaks in, so there's no template scan). WRITE-IF-CHANGED so an unchanged regen doesn't bump the mtime + cascade a recompile (the partial lands under the `templates` rerun-trigger); the generated `icons.html` is gitignored. The STL-viewer loading spinner is a self-contained `.stl-spinner` CSS ring (no icon, no dep on Tailwind scanning JS).
4. **Downloads the pinned fab-gui WASM editor bundle** (`FAB_GUI_VERSION` / `FAB_GUI_TAG` in `build.rs`, currently `0.16.0` / `web-v0.16.0`) from the `chotchki/fab-scad` GitHub release (asset `fab-gui-<version>.tar.gz`) into `$OUT_DIR/fab-gui` for `rust-embed` â€” the slicer/placer editor served under `/3d/editor` (Phase CW). fab-gui replaced the old fab-web/Bevy bundle (ONE codebase desktop + web; the OpenSCAD side-module is gone â€” scad-rs renders in the `geom/` Manifold worker now, so the bundle is `fab_gui.js` + `fab_gui_bg.wasm` + a `geom/` worker tree + a ROOT `libs.json`, no `openscad/`). Version-keyed by a sibling `fab-gui.version` marker so a bump re-downloads (same shape as the Tailwind CLI fetch), and it sha256-verifies EVERY file against the bundle's `manifest.json` (a corrupt/tampered download fails the build). Then it **drops each raw wasm ONLY when its `.gz` sibling ships** â€” `editor_asset` reconstructs identity by GUNZIPPING the `.gz` for a no-`Accept-Encoding` client (it can't brotli-decode), so a wasm that ships only `.br` MUST keep its raw or that client 500s (fab-web shipped the geom wasm `.br`-only; the conditional drop is safe-by-construction either way, guarded by `editor_geom_wasm_identity_for_no_encoding_client`). Exposes `FAB_GUI_VERSION` as a `rustc-env` so the runtime version-paths the editor's resource URLs (`/3d/editor/<ver>/â€¦`) for immutable cache-busting. **The pin is a hard gate:** `cargo build` 404s loudly if `FAB_GUI_TAG` doesn't match a published release, and after bumping the pin run `cargo clean -p hotchkiss-io` (the marker short-circuit caches the last fetch). The serving half is `web/features/three_d.rs` â€” a COOP/COEP-scoped `/3d/editor` document (cross-origin isolation confined to that route) rendered UNDER the real site nav (Phase CW.10: `templates/3d/editor.html` reuses the shared `partials/jumbotron.html` + `partials/nav.html` â€” the same header base.html renders, extracted to partials â€” in normal flow up top, then a full-dynamic-viewport tool region with CSS **scroll-snap** giving two rest states: the header (top) and the tool full-screen. A scroll jumps straight from one to the other, never resting in the clipped middle where the tool's own bottom STATUS BAR sat below the fold (the reported bug â€” the tool is exactly one viewport tall, so it only fully shows when the header is fully off; scroll-snap-align on `#site-header` + `#stage` + `scroll-snap-stop:always` skip the middle). The `#fab-gui` canvas the app binds to fills `#stage`; a boot splash lifts on the app's `fab-gui:ready` event). **Load-scroll gotcha:** the app focuses its canvas for keyboard input and the browser scrolls a focused element into view â€” which jumped the page ~276px past the header AT LOAD (reported on a real phone) â€” so the editor forces `preventScroll` on the canvas focus + snaps back to top through boot (`scrollRestoration=manual` + post-`ready` nudges); `editor_boots_in_browser` reproduces the jump if that guard regresses (asserts the page starts at top AND still pins on scroll). Every asset the page pulls (main.css, the nav's inline icons, self-hosted fonts) is SAME-ORIGIN so it survives COEP:require-corp â€” verified in-browser by `editor_boots_in_browser` (`crossOriginIsolated` stays true, and the sticky scroll pins the tool while the nav scrolls away). Bundle files serve from a `{*path}` wildcard that hands back the precompressed `.br`/`.gz` variant a client advertises (with the matching `Content-Encoding`, so it skips the site CompressionLayer â€” never double-compressed). Covered by `tests/three_d.rs` (serving) + `tests/e2e_browser.rs::editor_boots_in_browser` (real headless-Chrome boot: subresources load, canvas present, cross-origin isolated, splash lifts, no wasm error).
5. Re-runs only when `assets/scripts`, `templates`, `migrations`, or `build/svg-icons` change.

`assets/` and `templates/` are bundled into the binary at compile time via `rust-embed` (`web/static_content.rs`) and `askama` respectively â€” there are no loose static files at runtime besides the SQLite database.

## Runtime architecture

`src/main.rs` â†’ `lib::real_main` â†’ `tray-wrapper` (macOS menu-bar anchor) â†’ `create_server` closure â†’ `ServiceCoordinator`.

The coordinator (`src/coordinator/service_coordinator.rs`) wires four long-lived tokio tasks connected by `broadcast` channels:

```
IpProviderService â”€â”€IPsâ”€â”€â–¶ DnsProviderService (Cloudflare API)
                                 â”‚
AcmeProviderService â”€â”€RustlsConfigâ”€â”€â–¶ EndpointsProviderService (HTTP/HTTPS axum servers)
```

- **IpProviderService** polls `https://1.1.1.1/cdn-cgi/trace` hourly (parsing the `ip=` line) and broadcasts the public IP. Connecting to the IPv4 literal forces a v4 path. In `debug_assertions` builds it forces `127.0.0.1` so dev never touches the public DNS.
- **DnsProviderService** updates Cloudflare A records when the IP differs from DNS.
- **AcmeProviderService** loads or orders a Let's Encrypt cert (DNS-01 via Cloudflare), persists it in SQLite (`certificates` table), broadcasts a `RustlsConfig`, and refreshes every 6h. Serves **HTTP/2** (ALPN `h2`, h1.1 fallback; no HTTP/3) â€” the ALPN policy + the axum-server-needs-ALPN-or-it-silently-stays-h1 gotcha live in `build_server_config` and its unit test.
- **EndpointsProviderService** waits for a `RustlsConfig` then starts both an HTTPâ†’HTTPS redirect server on `:80` and the real axum app on `:443` (served with `into_make_service_with_connect_info::<SocketAddr>()` so handlers/middleware can see the client IP). It also owns the `tower-sessions-sqlx-store` session GC task, a daily `request_log` prune task (90-day retention), and a **daily DB backup task** (`coordinator/backup.rs`): each tick `VACUUM INTO`s a dated snapshot (`database-YYYY-MM-DD.sqlite`) under `Settings::backup_path`, then prunes to a rolling 7 most-recent. The backup loop never returns and matches+logs every fallible step (`tracing::error!`) so a backup failure can't bubble out and trip the coordinator's `try_join!` (which would take the whole app down) â€” a bad day self-heals on the next tick. Backups exist only on disk; Backblaze covers off-site.

If any task returns, `tokio::try_join!` fails the whole coordinator â€” there is no auto-restart.

## Configuration

`Settings` is loaded from JSON, in this order:

1. First CLI arg, if present (this is what `bacon` and dev use: `data/config.json`).
2. Otherwise `~/Library/Application Support/io.hotchkiss.web/config.json` on macOS, `$HOME/...` elsewhere.

Required fields: `cloudflare_token`, `domain`. Optional path fields â€” `database_path`, `log_path`, `cache_path`, `backup_path` â€” default to `~/Library/Application Support/io.hotchkiss.web/data/database.sqlite`, `~/Library/Logs/io.hotchkiss.web`, `~/Library/Caches/io.hotchkiss.web`, and `~/Library/Application Support/io.hotchkiss.web/backups` respectively when omitted (see `Settings::resolve` in `src/settings.rs`). `backup_path` is where the daily DB snapshots land (created on first backup if missing). The on-disk JSON is deserialized into a private `RawSettings`; unknown keys are silently ignored. The `data/` directory in the repo is gitignored and contains the dev config + SQLite database.

The macOS app is **not sandboxed** (the `com.apple.security.app-sandbox` entitlement was dropped, 2026-05) â€” `NSHomeDirectory()` returns the real home, so the paths above are literal. On the production Mac the app runs as a LaunchAgent (`build/macos/io.hotchkiss.web.plist`) in the user GUI session, which keeps the tray icon alive; macOS Mojave+ lets the non-root process bind ports 80/443 because axum binds `INADDR_ANY`.

## Things to watch out for

- **Vendored OpenSSL**: `openssl` is pinned with `features = ["vendored"]` to avoid system-OpenSSL packaging issues.
- **Ports default to 80/443 but are configurable** (`http_port` / `https_port` in `Settings`, since Phase 11; beta runs 8080/8443) â€” running locally as non-root on 80/443 will fail to bind, so set high ports in your dev config or accept the bind failure. (On the production Mac 80/443 work unprivileged because macOS Mojave+ allows non-root binds to ports <1024 when binding `INADDR_ANY`.)
- **`assets/styles/main.css` is generated** â€” never edit it; edit `styles/tailwind.css` and let `build.rs` recompile.
- **Pushing deploys (inverted flow, Phase 12)** â€” `git push origin main` rebuilds + restarts **beta**; only a `v*` **tag** push rebuilds + restarts **prod** (~10â€“15s downtime; the restart is a graceful `launchctl kill TERM` + `KeepAlive` relaunch â€” Phase BY â€” not a SIGKILL). A failed build aborts before the swap (the running app keeps serving), but a *successful build of broken runtime behavior* goes live on whichever target the ref maps to. `origin` is the mini (deploy); `github` is the public mirror, re-added 2026-06-29 after the Phase CM docs-scrub + history rewrite â€” push `main`/`v*` tags there manually to publish (nothing auto-pushes).
