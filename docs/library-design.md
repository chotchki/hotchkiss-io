# Role-gated sections — the family library (and everything that rides on it)

The family wants to listen to audiobooks we own, from their own devices. We own them — serving them publicly would be distribution, so no anonymous access to the pages OR the bytes. This doc designs the permission FOUNDATION (a new trust tier + minimum-role visibility on pages and media + role-scoped mutations) and its first consumer (a Library tab starting with audiobooks). A second consumer is already queued — a "Home" tab giving visiting family an Apple-Home-style device view — and it shapes the foundation (device control means non-admins issuing POSTs) but its smart-home integration is deliberately NOT designed here (its own SPEC when it's ready).

## The problem, concretely

Three gaps, none expressible with today's machinery:

1. **`Registered` is not trust.** Registration is open — any passkey holder becomes `Registered` (`login.rs:155`, plus the first-user CASE in `UserDao::create`). It means "has logged in", nothing more. The only role predicates in the tree are `is_admin()` and `is_authenticated()`; there is no tier for "my family".
2. **Every GET is public by design.** `require_admin_for_mutations` waves through all GET/HEAD/OPTIONS site-wide; the `/admin` nest is the single non-public-GET exception. A family-only page has nowhere to hang a read gate.
3. **Media bytes are guarded ONLY by URL unguessability.** `/media/file/<url_key>` takes no `SessionData` at all — the HMAC token is the whole control. Fine against scanners, useless once a link is shared or a gated page's HTML (which contains the URL) is saved.

## Threat model — what this stops (and deliberately doesn't)

Stops: anonymous visitors + crawlers, self-registered strangers (they cap out at `Registered`), and leaked byte-URLs (the link alone stops working — a role is required). Existence enumeration stays closed via the Phase-CU oracle rule (hidden = byte-identical cat-404).

Deliberately NOT DRM: a family member with access can save the file — that's the same trust as handing them the disc. No watermarking, no expiring media URLs, no per-device licensing. The threat is the public internet, not my mother.

One deliberate disclosure class: CODE-DEFINED `/library` routes tell an insufficient viewer to sign in (see the section below). Route names ship in the public source mirror anyway — they were never secret. What IS secret is the DATA: page slugs, titles, covers and bytes all stay miss-shaped.

## The role ladder

- New variant `Role::Family`, sitting between `Registered` and `Admin`. Role is TEXT-persisted via strum variant names with no CHECK constraint (`0001` — `app_role text`), so the variant is additive: NO users migration, and serde/strum/sqlx pick it up everywhere including `/test/login?role=Family`.
- **The Ord trap:** `Role`'s variants are declared alphabetically (`Admin, Anonymous, Registered` — `roles.rs:17`), so a derived `Ord` would rank `Admin < Anonymous`. The ladder is an explicit `rank()` — Anonymous 0 < Registered 1 < Family 2 < Admin 3 — with a `Role::iter()`-driven unit test pinning it (and a doc comment forbidding `derive(Ord)`).
- `AuthenticationState` grows `role() -> Role` (`Authenticated(u)` → `u.role`, everything else → `Anonymous`) beside `is_admin()`/`is_authenticated()`. Every viewer-role derivation goes through it.
- **Family is promotion-only.** Registration stays open → `Registered`; chris promotes at `/admin/users`. `set_user_role` already parses any variant (`RoleForm`), so the handler is nearly free — but the UI is binary by construction (`UserRow.is_admin`, Admin-else-Registered badge, one Promote/Demote pair — `users.html:31-56`) and gets reworked to show the real role + a three-way control. While in there: reject `role=Anonymous` as a target (it's a sentinel, not an account level; today the handler would accept it).
- The last-admin guard keys on `== Role::Admin` on both sides — demote-to-Family respects it with zero changes.

## Session expiry — fix the false premise first

The config SAYS `Expiry::OnInactivity(Duration::days(1))` (`router.rs:51`), but tower-sessions 0.14 only pushes the expiry forward when the session is WRITTEN (`modified || always_save`), and this app writes the session exactly once — at login. The extractor is read-only, `refresh_session_role` writes request extensions not the session. So every session dies 24 hours after login no matter how active the user is — for a listening family that's mom's book stalling on the lock screen at buffer-end every single day, with no visible player to show a prompt. Not an edge case to verify on a phone; deterministic today.

Fix (Phase CZ, before anything is built on sessions): `refresh_session_role` re-saves the session for AUTHENTICATED requests, making OnInactivity mean what it says. Cost: one `tower_sessions` UPDATE per authenticated request — family-rate traffic, fine; anonymous traffic is untouched (no session exists to save). Alternative if that write rate ever matters: throttle the touch (re-save only when the last save is >1h old). Passkey re-auth stays the fallback for a genuinely idle day — FaceID is cheap, daily-forever was the bug.

## Page visibility — a second axis on the CU machinery

Phase CU (scheduled publishing) already built the shape this needs: a single `is_visible_to()` predicate, hybrid Rust/SQL enforcement, and a mapped set of read paths. Min-role is a second clause on the same skeleton.

- **Storage:** `content_pages.min_role TEXT NULL` (NULL = public) — symbolic like `app_role`, not a rank integer that couples the DB to enum numbering.
- **Rust side:** `is_visible_to(viewer: Role)` replaces the bool:
  `(special_page || viewer == Admin || !is_scheduled()) && viewer.rank() >= min_role_rank`.
  KEY CHANGE: the special-page exemption now covers ONLY scheduling. Today `special_page` short-circuits the whole check (`content_pages.rs:541`) — the role clause must apply to special pages too, because the `library` special page itself is what gates the nav tab. The column decodes as `Option<String>` and parses FAIL-CLOSED: an unknown non-NULL value ranks as Admin-only, never public.
- **SQL side:** the paged trio (`count_children` + `find_children_{newest,ordered}_paged`) gains one AND clause: `(CASE WHEN min_role IS NULL THEN 0 WHEN min_role = 'Registered' THEN 1 WHEN min_role = 'Family' THEN 2 ELSE 3 END) <= ?viewer_rank`. NULL is the ONLY public spelling — the ELSE fails closed, so a value the binary doesn't know (manual DB edit, a future role after a rollback) hides content instead of leaking it. The count and fetch predicates stay byte-identical (the CU pager-desync rule); the existing parity test (`content_pages.rs:937`) grows a role dimension, and a `Role::iter()`-driven test pins the SQL CASE against `rank()` so the two ladders can't drift.
- **Rollback honesty:** a binary rollback to a pre-gate release ignores `min_role` entirely and serves gated pages publicly (deploys swap bundles, the DB stays). If gated content exists, don't roll back past the gate — or blank the rows first.
- **Threading:** every `is_visible_to` call site swaps bool → role (~12 sites, recon-mapped — including `/3d`'s `three_d.rs:96`, which post-dates the CLAUDE.md read-path list; the leak sweep confirmed the list is complete, e.g. `children_pages` renders only inside the admin-editor branch). Crawler-facing paths stay UNCONDITIONALLY Anonymous: feed (`feed.rs:193`), sitemap (`seo.rs:134/145/157`), `redirect_to_first_page`. Session-facing paths derive the viewer via `SessionData` → `auth_state.role()`: `get_page_path`'s ancestor scan (a gated ANCESTOR hides its whole subtree — children inherit for free), `show_post` + its sibling retain, home's band retain, resume, and `paginate()` (its `viewer_is_admin: bool` param becomes `viewer: Role`).
- **Nav:** `TopBar::create` currently filters `is_visible_to(false)` unconditionally and never sees the session (`top_bar.rs:24`). It gains a viewer-role param (~15 call sites, mechanical) so a Family session SEES the Library tab and Anonymous doesn't. Feed/sitemap stay anonymous-only — gated content never appears there for anyone (they're unauthenticated fetches; a per-viewer feed would also break the 304 validator).
- **Oracle rule holds:** insufficient role on a CONTENT page → the SAME `render_not_found` cat-404 as a genuine miss, at both existing gate sites. A 403 would be an existence oracle. (Code-defined `/library` index routes are the one exception — below.)
- **Editor:** a Visibility select (Public / Registered / Family / Admin-only) as the 5th cell of the metadata grid (beside Posted, `get_page.html:112`), riding `PutPageForm` → `update()` — which stamps `page_modified_date`, so the feed/sitemap validators bust when a page flips public↔gated. A "Family" badge renders wherever the Scheduled badge does (cards, editor header, reader) — admin-facing by construction.
- **Inherit-on-create:** a new child page defaults `min_role` to its parent's (`post_page_path` has the parent row in hand). The ancestor scan already hides the subtree, but the stamp protects any FUTURE listing surface that queries children directly (belt AND suspenders — the paged SQL checks the child's OWN row).
- `render_cache` needs zero changes: it's content-hash keyed and stays correct because gating is whole-page — never inject role-dependent markup into the rendered markdown, or the cache would cross-serve it.
- The d2 `/diagram/<hash>` route is honestly WEAKER than url_keys: its hash is unkeyed SHA-256 of the source (no server secret), so anyone who can reconstruct a gated page's exact d2 source can fetch the rendered SVG. Acceptable — audiobook pages won't carry diagrams and guessing exact source bytes is hard — but it's a stated acceptance, not an equivalence. The cheap hardening if it ever matters: HMAC-key the registry hash (one-line change, registry is process-lifetime).
- Cost note: `ContentPageDao`'s explicit `query_as!` select lists mean the new column touches ~8 query sites + the struct + test constructors, and needs a `cargo clean -p hotchkiss-io` after the migration for the sqlx macros to re-validate.

## Media visibility — gating the bytes

`media.min_role TEXT NULL`, same semantics (NULL-only-public, fail-closed parse), enforced at all three public routes (each is anonymous today):

- **`/media/file/<url_key>`** (bytes): the handler gains a `SessionData` extractor (the session layers already run outer to the router — it just works) plus the gate rank. NOT by widening `find_by_url_key` — that's `LIMIT 1` and would return one arbitrary owner's `min_role`. A NEW query keeps the variant fetch and computes the gate as a scalar aggregate over the whole url_key group:
  `SELECT v.*, (SELECT MAX(CASE WHEN m.min_role IS NULL THEN 0 … ELSE 3 END) FROM media_variant v2 JOIN media m ON m.media_id = v2.media_id WHERE v2.url_key = v.url_key) AS required_rank FROM media_variant v WHERE v.url_key = ?1 LIMIT 1` — returned as a thin wrapper, `MediaVariantDao` untouched.
- **Why MAX (strictest-wins):** `url_key` is deterministic in the sha and its index is deliberately NON-unique (`0013:42`) — the same bytes shared by two items resolve through `LIMIT 1` to an arbitrary row. Loosest-wins can leak silently (invisible failure); strictest-wins can only break a public embed (visible — chris notices and fixes). Unit test: two items sharing a sha, one NULL one Family → Family wins.
- **`/media/<ref>`** (download 302) and **`/media/embed/<ref>`** (player HTML): same check via the media row both already load. A denied embed returns the standard error-span at HTTP 200 (`render_media_embed`'s existing miss shape — indistinguishable from a bad ref); denied bytes/download → 404, matching an unknown key.
- **Cache-Control:** gated variants get `private, max-age=31536000, immutable` — content-addressing keeps browser caching safe (and seek-friendly); only shared caches are cut out. Public media keeps `public` (`media.rs:133`).
- **Authoring — the default matters more than the control.** The natural motion for a book page is dropping the m4b onto the GATED page's editor; if that mints a public item, the byte gate is silently defeated (exactly the invisible failure strictest-wins exists to avoid). So: `upload_media` accepts a `min_role` multipart field (the non-file-field match is the slot; `add_encode` needs no field — variants inherit the item's gate); **`editor-support.js` sends the current page's visibility as the default** (the value is already in the editor form); the `/admin/media` library page gets an explicit default-visibility control; `fab publish`'s API path sends nothing → public, which is correct for it. Every library card renders the visibility BADGE prominently — a public audiobook is one glance away from noticed. Per-card selector → `POST /admin/media/{id}/visibility`.
- **Covers:** no separate gate — a cover only surfaces on pages that passed the page gate. But the cover's byte URL is gated if its media is, so the authoring rule is: public page ⇒ public cover (a gated cover on a public card would render broken).

## Role-scoped mutations — the seam the Home tab needs

`require_admin_for_mutations` stays THE single authz layer, gaining one table: exact-match `(Method, path, minimum Role)` entries checked by `rank()`, e.g. `(POST, "/library/progress", Family)`. Decision order: safe methods pass → anonymous WebAuthn allowlist → role-scoped allowlist → `is_admin()` fallback. Still default-DENY, still zero per-handler authz.

Two conventions make this the Home-tab seam, not just a one-off: **resource identifiers ride the request BODY** (as `/library/progress` does with `media_ref`) so entries stay exact-match — no path patterns; and **per-resource authorization beyond the coarse role gate lives in the handler** (the progress handler re-checks the media's own `min_role`) — which is exactly where "lights yes, door locks no" per-device granularity will live. It is NOT a generic ACL system; entries are code, reviewed like the WebAuthn ones.

## Audio — a first-class media kind

- Today an m4b silently ingests as `MediaKind::File` (a download button, no player): `parse_ffprobe` requires a `codec_type=="video"` stream and the failure degrades to `generic_file` (`probe.rs:106,151`). Zero audio rows are expected on prod (none were ever uploaded) — if any exist, re-upload; dedup makes it free. NO backfill.
- **`MediaKind::Audio`** (`"audio"` — the kind column is unconstrained text). Probe changes: add `-show_chapters` to the ffprobe args (`probe.rs:93`); classify by the first AUDIO stream when there's no video stream OR every video stream is embedded cover art — an m4b/mp3 with a cover ships an mjpeg/png "video" stream flagged `disposition.attached_pic`, so `FfStream` grows `disposition`.
- **Codec map is UNIVERSAL-only:** aac → `audio/mp4`, mp3 → `audio/mpeg`, flac → `audio/flac`. That's it. Opus, Vorbis and ALAC do NOT play across the stated mixed-device audience (Ogg never in Safari; ALAC only in Safari) — they stay `MediaKind::File` (download button, today's behavior) rather than minting a player that errors for half the family. **AAC m4b is the canonical format**; the doc says so, the library UI badge makes a File-typed audio upload visible at a glance. `duration_ms` comes free from `format.duration` (column already exists).
- **Chapters:** `media.chapters TEXT NULL` (JSON `[{start_ms, title}]`), stamped at ingest for Audio.
- **Embed** (new `render_embed_html` arm, cloning the STL arm's shape): `<audio controls preload="metadata">` with the `audio/*` sources, a download button for the largest variant, and `data-chapters`/`data-ref` for the player. Degrades to the bare native element without JS.
- **The player** (first-party, vendored-only like every other script): chapter list + seek, ±30s skips, playback rate, MediaSession metadata (lock-screen controls + cover art), resume — from **localStorage in this phase** (per-device; the server position swap is the progress phase's job, keeping that phase genuinely deferrable). Resume seek applies at `loadedmetadata` AND re-asserts on first `play` (iOS silently drops a `currentTime` set before metadata; a `#t=<seconds>` media fragment is the fallback WebKit honors natively); never autoplay after the seek.
- **request_log: exclude `/media/file/`.** Streaming means hundreds-to-thousands of range requests per listen, and greylist R3 fires at 1000 requests/IP/24h (calibrated on ~366-request human days) — two listeners behind one home NAT would plausibly greylist their OWN IP, and the rows would swamp the Humans audience + top-paths signal CQ/CY built. Media bytes are static-asset-class traffic, not content signal (covers/responsive images already spray this path on every page view). Cost stated honestly: no byte-route analytics. Mirrors the existing `/challenge` + `/admin/*` self-feed exclusions (`request_log.rs:51`); note the decision in the greylist design doc's deferred-levers section.
- **On-phone verification checklist (the 10.13 pattern — DD exits through it):** screen-off/background playback in a Safari tab AND an installed PWA (the home-screen app has its OWN cookie jar — separate login; pick and WRITE DOWN the supported mode after this test); MediaSession lock-screen controls; artwork with a GATED cover (WebKit's out-of-page artwork fetch may not carry credentials — fallback: credentialed in-page fetch → blob URL); resume position survives a cold-load first play; range seeking against `ServeFile`. Prereqs: a beta test page holding a gated embed + a Family-promoted beta user.
- No transcode ladder: the uploaded m4b IS the stream. `add_encode` accepts a hand-made low-bitrate AAC as a second variant if cellular listening ever hurts.

## The Library section itself

- A migration seeds `('library', '/library', -1, true)` mirroring `0023`, and stamps `min_role='Family'` on the row. That ONE row buys: the nav tab (Family+ only via the role-aware TopBar), the `/pages/library` redirect, ancestor-gating of the entire subtree and undeletability — all existing special-page machinery.
- **Routes** (the `/3d` shape — the module owns only the index; details stay on `get_page_path`): `/library` renders section doors from its children (AudioBooks now; manga/video are just new child pages later — zero new plumbing). `/library/audiobooks` is paginated book cards via `listing.rs::paginate` + the search/pager partials (built for exactly this).
- **The sign-in gate covers every CODE-DEFINED `/library` route** — `/library`, `/library/audiobooks`, future index routes — for an insufficient viewer: route names are in the public source, so a miss-shape buys nothing, and a session-expired bookmark that cat-404s is a support call from mom. Copy is state-aware: logged-out → "sign in" (+ the login link with `?next`); authenticated-but-insufficient (a self-registered stranger — they exist by construction) → a neutral "this section is restricted", no tier names, no sign-in loop. Everything DATA-defined (book pages under `/pages/library/...`, media) stays miss-shaped. One wrinkle the nav forces: tabs link `/pages/library` (`base.html:79`) and `get_page_path` runs its visibility scan BEFORE the special-page redirect — so for a special LEAF whose only failing constraint is ROLE (not schedule), issue the redirect anyway and let the target route show the gate. Both gate responses join the oracle tests.
- **Greylist interaction:** `/login` is already toll-exempt; add the `/library` INDEX (exact path, not the subtree) to `EXEMPT_PREFIXES` with the same rationale — a greylisted logged-out family member must reach the sign-in gate, and the gate page serves no content worth scraping.
- **Login `?next` — designed against the real flow** (the ceremony is `fetch()`-driven; a hidden form field has nowhere to ride, and `htmx-webauthn.js` hardcodes `window.location.href = "/"` at :161/:197): `login_page` (and `get_auth_opts`/`start_register`) validates `?next` and stashes it in the SESSION alongside the ceremony state; `finish_authentication`/`finish_registration` pop it and redirect there; the JS navigates to the fetch's final `response.url` instead of the literal `/`. Validation is strict: must start with `/`, second char neither `/` nor `\`, and NO backslash anywhere (`/\evil.com` is the classic WHATWG-parser open-redirect bypass — browsers treat `\` as `/`); both bypass strings are unit-test vectors.
- **A book page is a plain content_page:** `page_title`, gated cover, markdown (author / narrator / blurb) + the `![](/media/<ref>)` audio embed. Authoring flow: upload the m4b at `/admin/media` (or drop it on the gated page's editor — the default-visibility rule makes that safe) → copy `![]()` → paste.
- **Home stays quiet:** `show_home`'s hardcoded `["blog", "projects"]` is untouched — the public front door doesn't advertise the family section (mirrors `/3d`'s deliberate absence from home).

## Listening progress — resume where you left off

- `playback_progress(user_id, media_id, position_ms, updated_at)`, PK `(user_id, media_id)`; `media_id REFERENCES media(media_id) ON DELETE CASCADE` (matching `media_variant` — `MediaDao::delete_by_id` is a bare DELETE); user rows wiped in `delete_user`'s tx alongside `api_keys`.
- `POST /library/progress {media_ref, position_ms}` through the role-scoped allowlist (Family); the handler re-checks the media's own min_role (defense in depth) and upserts. `GET /library/progress/<media_ref>` returns the SESSION user's row only — and responds `Cache-Control: no-store` (a cached position would quietly defeat the phone→iPad handoff this exists for; the player fetch uses `cache: 'no-store'` too).
- Saves: throttled ~30s + on `pause` + on `visibilitychange→hidden` + a `pagehide` sendBeacon — the throttle is the real guarantee on iOS (a locked tab jettisoned by the OS never sees pagehide; the beacon is best-effort). "Progress saves continue with screen off" joins the phone checklist. The player swaps its localStorage resume for the server position, keeping localStorage as offline fallback; a progress-save rejection means "session expired → prompt sign-in", never a silent stall.
- Exclude `/library/progress` from `request_log` (machine telemetry at ~120 rows/listening-hour, the same self-feed logic as `/challenge`) — also keeps it out of greylist R3's per-IP total.
- Cross-device resume (phone → iPad) is the point; the phase is genuinely deferrable now that the player ships with localStorage resume.

## What this deliberately does NOT do

- No DRM or watermarking — family trust, not adversarial users.
- No TTS-from-text-EPUBs — "listening to ebooks" means audio files we own. A text-ebook reader, a manga (CBZ) reader and video sections are future `/library` children that reuse the gate unchanged.
- No invite links or closed registration — open registration → `Registered` stays; a registration flood still lands at zero access.
- No per-API-key role scoping — keys stay full-delegation; `/admin/api-keys` stays admin-only.
- No audio transcoding — AAC m4b in, AAC m4b out; non-universal codecs stay download buttons.
- No Home-tab design — the foundation (Family + role-scoped mutations + the ids-in-body/handler-check conventions) is sized for it; the HomeKit/Matter side is its own SPEC.

## Beta caveat

Everything carries by default: users (Family promotions land on beta — family passkeys work there via the shared rp_id), both `min_role` columns, `playback_progress` (harmless — beta is overwritten on every `main` push). `crypto_keys` id 2 is preserved so url_keys resolve, and the gate is ROLE-based, so beta enforces identically to prod. Audiobook files over 2 GiB never rsync (`--max-size=2G`) — their variant rows still land, so beta 404s those bytes (typical m4bs are 100–500 MB; fine). Every phase below names its beta validation step — the gate is exactly the kind of thing to prove on beta before a prod tag.
